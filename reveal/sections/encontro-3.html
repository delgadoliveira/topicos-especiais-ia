🧠 Parte 1: O desafio do contexto

Um LLM lê tudo a cada chamada — e paga por cada token lido.
Quando a conversa cresce, três coisas acontecem:

💸 Custo explode
🐢 Latência sobe
🧠 Qualidade cai
Vamos entender por quê — e as soluções que o mercado usa.

A janela é grande, mas não é infinita — e cada token custa.

⚠️ Os 3 problemas da context window:
  1. Limite duro — passou, a API rejeita.
  2. Custo — você paga por TODOS os tokens, toda chamada.
  3. "Lost in the middle" — modelos esquecem info no meio do contexto longo (Liu et al., 2023).

Lost in the Middle — o efeito U

85%Início
60%25%
45%Meio
55%75%
80%Fim

Onde o modelo presta atenção

Instruções importantes no começo e pergunta final no fim.

Onde o modelo falha

Informação crucial enterrada no meio do contexto longo.

Em contextos longos (>10k tokens), modelos prestam mais atenção ao início e fim, e tendem a "esquecer" o meio. Por isso colocar instruções importantes no começo e a pergunta no fim ajuda.
🧠 Explicação rápida: Modelos tendem a ignorar informação no meio de contextos longos. As primeiras e últimas posições recebem mais atenção.
📌 Na prática: coloque informação importante no INÍCIO ou no FIM do contexto, nunca no meio.
📚 Liu et al. (2023) — "Lost in the Middle: How Language Models Use Long Contexts"

3.2 Estratégias de context management

O contexto é o recurso mais precioso do agente — e é finito. Gerenciá-lo bem é a diferença entre um agente que funciona e um que "esquece" ou estoura custos.

Conversa cresce
→
Tokens > limite
→
Estratégia de gestão
💡 Princípio: nunca envie tudo — envie apenas o que o modelo precisa agora para responder bem.

3.2 As 4 estratégias principais

✂️ Truncamento (Sliding Window)
Mantém apenas as últimas N mensagens e descarta o resto.
  • Quando usar: chatbots simples, suporte ao cliente com conversas curtas
  • Implementação: messages = messages[-10:]
  • Risco: perde contexto antigo que pode ser crucial ("lembra que eu disse...")
📦 Usado por: ChatGPT free (antes do Memory), bots de atendimento
📝 Sumarização progressiva
Comprime mensagens antigas em um resumo gerado pelo próprio LLM.
  • Quando usar: conversas longas onde o contexto histórico importa
  • Implementação: a cada N msgs, gera resumo e substitui as antigas
  • Risco: +1 chamada LLM (custo extra), pode perder detalhes numéricos
📦 Usado por: LangChain ConversationSummaryMemory, Claude Projects
🔍 RAG (Retrieval-Augmented Generation)
Busca apenas os trechos relevantes de uma base de conhecimento externa.
  • Quando usar: base de documentos grande (manuais, código, wikis)
  • Implementação: embed query → busca vetorial → top-k docs → injeta no prompt
  • Risco: depende da qualidade do retriever e dos chunks
📦 Usado por: Perplexity, Notion AI, GitHub Copilot, Glean
🗄️ Delegação (Sub-agentes / Handoffs)
Sub-agente processa contexto pesado e devolve apenas a síntese.
  • Quando usar: tarefas que exigem contextos incompatíveis (código + docs + dados)
  • Implementação: agente A envia tarefa ao agente B, recebe resposta compacta
  • Risco: orquestração mais complexa, latência de múltiplas chamadas
📦 Usado por: Devin, OpenAI Swarm, CrewAI, AutoGen

Exemplo: sumarização automática

📖 Em palavras: conte os tokens; se passar do limite, preserve as últimas 4 mensagens e resuma o restante em 200 palavras.
# 📋 Execute no Google Colab — usa HuggingFace (token gratuito)

def count_tokens_approx(messages: list) -> int:
    """Conta tokens de forma aproximada (1 token ≈ 4 chars em PT-BR)."""
    return sum(len(m["content"]) // 4 for m in messages)

def summarize(messages: list) -> str:
    """Sumariza mensagens usando HuggingFace Inference API."""
    texto = "\n".join(f"{m['role']}: {m['content']}" for m in messages)
    resp = client.chat.completions.create(
        model="Qwen/Qwen2.5-7B-Instruct",
        messages=[{"role": "user", "content": f"Resuma em 200 palavras:\n{texto}"}],
        max_tokens=250,
    )
    return resp.choices[0].message.content

Exemplo: sumarização automática — continuação

def manage_context(messages: list, max_tokens: int = 8000):
    """Se passar do limite, sumariza mensagens antigas e preserva as recentes."""
    if count_tokens_approx(messages) < max_tokens:
        return messages

    keep = messages[-4:]
    to_summarize = messages[:-4]
    summary = summarize(to_summarize)  # ← chamada real ao HuggingFace!
    return [{"role": "system", "content": f"Resumo do histórico anterior:\n{summary}"}, *keep]

# --- Teste ---
historico = [
    {"role": "user", "content": "Quero analisar vendas do Q1"},
    {"role": "assistant", "content": "Claro! Vou buscar os dados..."},
    {"role": "user", "content": "Agora compare com Q2"},
    {"role": "assistant", "content": "Q1: R$2M, Q2: R$2.8M — crescimento de 40%."},
    {"role": "user", "content": "Projete Q3"},
    {"role": "assistant", "content": "Com base na tendência, Q3 ≈ R$3.2M."},
] * 50  # Simula conversa longa (300 mensagens)

resultado = manage_context(historico, max_tokens=500)
print(f"Original: {len(historico)} msgs → Comprimido: {len(resultado)} msgs")
print(f"Resumo: {resultado[0]['content']}")

Hands-on: RAG completo com Chroma

☁️ 100% gratuito: usa embeddings locais (sentence-transformers) e o modelo Qwen 2.5 via HuggingFace Inference API (grátis). Sem chave paga!
from langchain_community.document_loaders import TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_huggingface import HuggingFaceEmbeddings, HuggingFaceEndpoint
from langchain_chroma import Chroma
from langchain.chains import create_retrieval_chain
from langchain.chains.combine_documents import create_stuff_documents_chain
from langchain_core.prompts import ChatPromptTemplate

# 1. Carregar e dividir o documento
docs = TextLoader("manual.txt").load()
chunks = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=50
).split_documents(docs)

# 2. Criar embeddings LOCAIS (gratuito, roda na sua máquina)
embeddings = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2")
vstore = Chroma.from_documents(chunks, embeddings)
💡 all-MiniLM-L6-v2: modelo de 80MB, rápido, bom para português. Alternativa: intfloat/multilingual-e5-small para melhor cobertura multilíngue.

Hands-on: RAG completo com Chroma — continuação

# 3. Configurar retriever (busca os 4 chunks mais similares)
retriever = vstore.as_retriever(search_kwargs={"k": 4})

# 4. Criar o prompt com contexto injetado
prompt = ChatPromptTemplate.from_template(
    "Responda baseado APENAS no contexto:\n\n{context}\n\nPergunta: {input}"
)

# 5. Usar Qwen via HuggingFace (gratuito)
llm = HuggingFaceEndpoint(
    repo_id="Qwen/Qwen2.5-7B-Instruct",
    huggingfacehub_api_token="hf_SEU_TOKEN_GRATUITO",  # huggingface.co/settings/tokens
)
combine = create_stuff_documents_chain(llm, prompt)
chain = create_retrieval_chain(retriever, combine)

resultado = chain.invoke({"input": "Como configuro o produto?"})
print(resultado["answer"])
✅ O que esperar: a resposta cita e usa o conteúdo do manual.txt. Se a info não está no arquivo, o prompt instrui o agente a admitir que não sabe.
⚠️ Limitações do modelo gratuito: rate limit (~5 req/min), contexto menor (~4K tokens). Para produção: use modelo local com Ollama ou API paga.

Padrão 1 · Query Rewriting

A pergunta do usuário raramente é uma boa query de busca.

❌ Pergunta do usuário
"e o problema que falei ontem, ainda tá acontecendo?"
→ vetor sem contexto, retrieval inútil
✅ Query reescrita pelo LLM
"erro 504 timeout no endpoint /api/login mencionado em 03/11/2025"
→ vetor rico, retrieval certeiro 
def reescrever_query(historico, pergunta):
    return llm.invoke(f"""
    Histórico: {historico}
    Pergunta atual: {pergunta}
    Reescreva como query de busca standalone, com todos os termos técnicos.
    """)

Padrão 2 · Multi-Query & HyDE

🔁 Multi-Query
Gere N reformulações da pergunta e busque com todas para aumentar recall.
queries = llm.invoke(f"""
Gere 4 reformulações da pergunta:
{pergunta}
"""
).split("\\n")
docs = []
for q in queries:
    docs += vectordb.search(q, k=5)
docs = dedupe(docs)
📈 Na prática, costuma melhorar o recall em 20–40%.

Padrão 2 · Multi-Query & HyDE — continuação

🎭 HyDE (Hypothetical Document Embeddings) cria uma resposta hipotética e usa o embedding dela para buscar.
hipo = llm.invoke(f"""
Escreva um trecho que responderia:
{pergunta}
"""
)
docs = vectordb.search(hipo, k=5)
💡 Funciona porque “resposta → resposta” costuma ficar mais perto no espaço vetorial que “pergunta → resposta”.

Search híbrido — porque puro vetor não basta

Busca semântica é boa com conceitos, mas falha em termos exatos: IDs, datas, códigos, siglas. A solução: combinar semântico + lexical.

Pergunta
→
Busca semântica
+
Busca lexical
BM25 / full-text
→
Merge + rerank
→
Top-K final
Semântico
Entende conceitos e sinônimos; falha em termos exatos, números e códigos.
Lexical (BM25)
Acerta termos exatos e IDs; não entende paráfrases.
🥇 Reranker: reordena os top-20 com um cross-encoder (Cohere Rerank, BGE-reranker) e costuma melhorar a resposta final.

🔗 Recuperar não basta — precisamos de atribuição

RAG trouxe informação externa para o contexto do LLM. Mas um problema permanece:

Como saber se a resposta veio dos documentos ou foi inventada?

→ É aqui que entra o Grounding: a técnica de ancorar cada afirmação a uma fonte verificável.

🏢 Caso real: Air Canada (2024) perdeu processo porque chatbot inventou política de reembolso inexistente. Grounding teria exigido citação do documento-fonte.

🎯 Grounding — o conceito

Sem grounding

o agente inventa fatos, cita fontes falsas e responde com confiança errada.

Com grounding

ele busca informação real antes de responder e cita fontes verificáveis.

💡 RAG, busca web e bancos de dados são técnicas de grounding: conectam o LLM ao mundo real.

O que é Grounding?

Grounding = ancorar cada afirmação do modelo em uma fonte verificável.
Toda resposta precisa responder: "de onde isso veio?"
❌ Sem grounding
"O contrato exige aviso prévio de 60 dias."
→ Verdade? Inventado? Impossível auditar.
✅ Com grounding
"O contrato exige 60 dias de aviso prévio [Cláusula 14.3, contrato_v3.pdf]."
→ Auditável, rastreável, defensável.
📚 Leitura essencial: Rashkin et al. (2023) — "Measuring Attribution in NLG Models" formaliza o conceito AIS (Attributable to Identified Sources). Bohnet et al. (2022) — "Attributed Question Answering" propõe o benchmark AQA. Em jurídico, médico e financeiro, grounding deixou de ser opcional — virou compliance.

Por que grounding falha

Pergunta
→
Retrieval
→
Contexto
100 chunks
→
LLM gera resposta
❌ Cita chunk errado
❌ Mistura fatos
❌ Inventa citação
❌ Ignora contexto
Mesmo com RAG, a "grounding rate" (% de afirmações ancoradas) raramente passa de 70-80% sem técnicas dedicadas.
⚖️ Exemplo real: Em 2023, um chatbot jurídico nos EUA citou casos inventados (Mata v. Avianca). O advogado foi multado por submeter citações fabricadas por IA ao tribunal.

Técnicas de grounding (1/2)

1. 🆔 Inline citations com IDs únicos
Cada chunk recebe [doc_id:chunk_id]. Prompt pede citação após cada frase.
2. 📋 "According to X" prompting
Forçar prefixo "Segundo [fonte X]…" reduz alucinação em 20% (Weller et al., 2023).
3. 🚪 Refusal explícito
Prompt: "Se o contexto não contém a resposta, responda 'Não encontrei nos documentos fornecidos.'"
4. 🔍 Contextual grounding check
Após gerar, um segundo LLM verifica: "Cada afirmação é suportada pelo contexto?"

Técnicas de grounding (2/2) — em código

📖 Camada 1: force o LLM a responder só com base nos documentos e a citar a fonte de cada afirmação.
SYSTEM = """Responda APENAS com base nos documentos abaixo.
Para CADA afirmação, adicione [doc_id] da fonte.
Se não souber, responda exatamente: "Não encontrei nos documentos."
NÃO use conhecimento externo."""

contexto = "\n".join([f"[doc_{d.id}] {d.content}" for d in docs_retrieved])
resposta = llm.invoke(f"{SYSTEM}\n\n{contexto}\n\nPergunta: {q}")

Técnicas de grounding (2/2) — continuação

📖 Camada 2: rode uma verificação automática que classifica cada frase como suportada, parcial ou não suportada.
verificacao = llm.invoke(f"""
Resposta: {resposta}
Contexto disponível: {contexto}
Para cada afirmação na resposta, marque:
- SUPORTADA (existe no contexto)
- NÃO_SUPORTADA (alucinação)
- PARCIAL (parcialmente apoiada)
JSON: {{"afirmacoes": [{{"texto": ..., "status": ...}}]}}
""")
🎯 Monitore Groundedness, Citation accuracy e Answer relevance.

🧬 Synthesis

🧬
1. Retrieve
Buscar chunks relevantes
2. Augment
Montar prompt com contexto
3. Generate
LLM sintetiza a resposta final
Boa synthesis não copia chunks: ela integra, compara e contextualiza a informação recuperada.

O problema da Synthesis

Recuperei 20 chunks de 5 fontes. E agora?

📄 Doc A diz: "X é 30%"
📄 Doc B diz: "X é 40%"
📄 Doc C diz: "X cresceu em 2024"
📄 Doc D contradiz Doc A
📄 Doc E é tangencial
🧬 Synthesizer → resposta única, coerente, citando conflitos
Synthesis ≠ Retrieval. É um passo de raciocínio que requer: dedup, resolução de conflito, hierarquização, citação atribuída.

Padrão 1 · Map-Reduce sobre documentos

Para muitos docs ou docs grandes — não cabem todos no contexto.

Doc 1
Doc 2
Doc N
↓
Map: extrai info relevante
→
Reduce: combina, resolve conflitos, responde
Map: rodar em paralelo, barato (modelo menor). Reduce: modelo maior, com instruções de citação.
Variante: Refine — atualiza resposta incrementalmente conforme lê cada doc.

Padrão 2 · Resolução de conflitos

Quando fontes discordam, o agente não pode simplesmente escolher uma.

SYNTH_PROMPT = """Sintetize a resposta usando os trechos abaixo.
REGRAS:
1. Se fontes CONCORDAM, afirme com confiança e cite todas.
2. Se fontes DISCORDAM, apresente as visões DIVERGENTES explicitamente:
   "Segundo [A], X = 30%. Já [B] reporta X = 40%."
3. Se a divergência tem causa identificável (data, escopo), explique:
   "A diferença pode se dever a [A] ser de 2023 e [B] de 2024."
"""

Padrão 2 · Resolução de conflitos — continuação

SYNTH_PROMPT = """
4. NUNCA invente uma 'média' entre fontes contraditórias.
5. Cite com [doc_id] após cada afirmação.
Trechos:
{contexto}
Pergunta: {pergunta}
"""
⚠️ Anti-padrão clássico: "média de alucinações" — pegar 3 números diferentes e dar a média. Pior que escolher um.

Para corpora gigantes (livros, repositórios, milhares de tickets):

citando origens

🧪 Context Engineering — o novo nome do jogo

Prompt engineering → Context engineering
📦 O que entra
System, exemplos, retrievals, memória, tool results, histórico — em qual ordem?
🗑️ O que sai
Compressão, summarization, eviction policies e mitigação de “lost in the middle”.
🎚️ Quanto e quando
Dynamic context: recuperar só quando precisa e gastar tokens onde importa.
🔁 Como evolui
O contexto muda turn a turn; ele não é estático.
📚 Termos atuais: Context engineering (Karpathy), Context rot (Anthropic) e Context economy.

Tipos de memória de longo prazo

Inspirados na psicologia cognitiva, agentes modernos implementam 3 tipos de memória persistente — cada um serve um propósito distinto:

📖 Episódica
"O que aconteceu"
  • Eventos, conversas passadas, ações tomadas
  • Permite ao agente dizer: "da última vez que falamos sobre X, você decidiu Y"
  • Armazenada como sequência temporal
📦 Produto: ChatGPT Memory, Claude Projects, Copilot Memory
🧪 Técnica: log de interações indexado por timestamp + embedding
🧠 Semântica
"O que sei sobre o mundo/usuário"
  • Fatos, preferências, contexto persistente
  • Permite personalização: "usuário é vegano e mora em SP"
  • Armazenada como pares chave-valor ou knowledge graph
📦 Produto: Mem0, Zep knowledge graphs, perfil do usuário
🧪 Técnica: extração automática de fatos + deduplicação
⚙️ Procedural
"Como fazer as coisas"
  • Habilidades aprendidas, scripts, workflows
  • Permite ao agente melhorar: "aprendi que para deploy uso o script X"
  • Armazenada como templates ou regras executáveis
📦 Produto: Claude Skills, Cursor Rules, Custom GPTs
🧪 Técnica: prompt templates versionados + retrieval por similaridade
🧠 Analogia humana: Episódica = seu diário · Semântica = sua enciclopédia pessoal · Procedural = seu caderno de receitas

Frameworks de memória persistente

O mercado oferece soluções prontas para dar memória ao seu agente — de plug-and-play a custom:

🧠 mem0
Layer de memória plug-and-play — detecta automaticamente o que vale lembrar.
  • Como funciona: extrai fatos das conversas, deduplica, e retorna os relevantes via search()
  • Setup: pip install mem0ai → 3 linhas para funcionar
  • Diferencial: memória automática (não precisa dizer "lembre disso")
  • Ideal para: MVPs, protótipos, agentes pessoais
⚡ Zep
Open source. Knowledge graph temporal + vector store. Foco em produção.
  • Como funciona: constrói grafo de conhecimento temporal sobre o usuário
  • Setup: Docker + API REST — getzep.com
  • Diferencial: relaciona fatos entre si ("João é gerente de Maria")
  • Ideal para: aplicações empresariais multi-tenant, compliance
🔄 LangGraph + Checkpoints
Persiste o estado completo do agente (incluindo memórias) em Postgres/Redis.
  • Como funciona: cada nó do grafo pode ler/escrever memórias no state
  • Setup: MemorySaver() ou PostgresSaver()
  • Diferencial: controle total — você decide exatamente o que persistir
  • Ideal para: agentes complexos com estado custom, workflows longos
📚 Letta (ex-MemGPT)
Paper seminal "OS para agentes" — memória hierárquica inspirada em sistemas operacionais.
  • Como funciona: context window = "RAM", banco externo = "disco". Agente faz page-in/page-out
  • Setup: pip install letta → servidor local com UI
  • Diferencial: memória virtualmente infinita, auto-gerenciada
  • Ideal para: pesquisa, agentes de longa vida, conversas de meses
📌 Como escolher? Protótipo rápido → mem0 · Produção com grafos → Zep · Controle total → LangGraph · Pesquisa/memória infinita → Letta

Exemplo: memória com mem0

📖 Em palavras: o mem0 funciona como um caderno do usuário: você grava fatos com add() e recupera só as memórias relevantes com search().
from mem0 import Memory
m = Memory()
user_id = "alan"
m.add("Prefiro respostas em português, tom técnico", user_id=user_id)
m.add("Trabalho com Python e TypeScript", user_id=user_id)
m.add("Não gosto de emojis em código", user_id=user_id)

Exemplo: memória com mem0 — continuação

def chat(pergunta: str):
    memorias = m.search(pergunta, user_id=user_id, limit=5)
    contexto = "\n".join(m["memory"] for m in memorias["results"])
    # Mock de resposta (em produção: chamaria LLM real)
    resposta = f"[Com contexto de {len(memorias['results'])} memórias] "

🧠 Além do RAG: memória como arquitetura

No Stanford CS329A, a ideia não é só “colocar mais texto no prompt”.
É tratar contexto como um sistema de memória, com camadas e políticas.

🪟 Contexto = mesa de trabalho
O que está aberto agora, visível para o modelo.

Analogia: sua mesa ou aba atual do navegador.

Se você lota a mesa com tudo, fica caro, lento e confuso.

🗃️ Memória externa = armário
Informações ficam guardadas fora do prompt.

Pode ser Vector DB, banco relacional, arquivos, sumários ou checkpoints.

O agente busca só o que precisa, em vez de carregar tudo toda hora.

⚖️ Política de memória
Quem entra, quem sai e quem volta para o contexto.

Perguntas-chave: o que vale lembrar? o que pode ser resumido? o que deve ser reconsultado?

Isso é context engineering na prática.

📚 Fonte-guia: Stanford CS329A — memória, context management e agentes que escolhem como usar contexto.

🧠 MemGPT — LLMs como sistemas operacionais

MemGPT (Packer et al., 2023) propõe olhar para o agente como um OS:
contexto principal = RAM · memória externa = HD.

Usuário fala
→
Contexto ativo
“RAM”
↔
Memória arquivada
“HD / disco”
→
Resposta
💬 O que fica na “RAM”
Pergunta atual, instruções e poucos fatos críticos.

É a parte cara e limitada: tudo aqui consome tokens a cada chamada.

Por isso o agente não deve deixar “tudo aberto”.

💾 O que vai para o “HD”
Histórico antigo, detalhes raros e documentos grandes.

Paging out: tirar do contexto atual, mas guardar para recuperar depois.

Paging in: trazer de volta quando a tarefa exigir.

🧑‍💻 Quem decide?
O próprio agente gerencia a memória.

Na visão do MemGPT, o LLM recebe instruções explícitas para decidir o que manter, resumir ou arquivar.

É menos “prompt gigante” e mais “gerente de memória”.

🖥️ Analogia simples: seu computador não coloca todos os arquivos na RAM; o agente bom também não.

⚡ CacheBlend & Efficient RAG

Para RAG funcionar em produção, não basta recuperar bem — precisa ser rápido, barato e repetível.

♻️ Reaproveitar o que já foi lido
Não recalcule tudo do zero a cada pergunta parecida.

Exemplo: política de RH, manual da empresa e docs da API aparecem o dia inteiro.

Se esses blocos já foram processados, faz sentido cacheá-los.

🧩 CacheBlend = fusão eficiente
Mistura pedaços recuperados com cache, reduzindo recomputação.

A ideia central é: reusar conhecimento recuperado e recalcular só o mínimo necessário.

Resultado típico: menor time-to-first-token e mais throughput.

🏭 Efficient RAG na prática
Caching + híbrido + rerank + compressão.

RAG escalável normalmente combina: cache de consultas frequentes, busca híbrida, reranker e compressão de contexto.

Objetivo: dar resposta boa sem pagar a conta de um prompt monstro.

📚 Stanford CS329A enfatiza isso: retrieval útil precisa virar knowledge fusion prática, não só demo bonita.

🧩 Cartridges — conhecimento portátil

Cartridges são representações leves de contexto longo: o agente “estuda” um domínio antes e guarda um pacote reutilizável para carregar sob demanda.

📚 1. Self-study offline
O agente lê docs, código ou um corpus inteiro antes do uso.

Em vez de reprocessar 500 páginas sempre, ele faz uma fase de estudo e destilação.

É como montar um resumo inteligente com estrutura para consulta futura.

🎮 2. Vira um “cartucho”
Conhecimento compactado, leve e reutilizável.

Analogia: videogame antigo — você encaixa o cartucho certo e ganha aquele mundo.

No agente, o cartucho carrega fatos, estrutura, índices e atalhos daquele domínio.

🚀 3. Load on demand
Só entra no contexto quando o tema pede.

Se a pergunta é sobre jurídico, carrega o cartucho jurídico; se é sobre código, carrega o cartucho do repositório.

Menos contexto inútil, mais foco.

💡 Por que isso importa?
Cartridges tentam unir o melhor de dois mundos: profundidade de contexto e baixo custo de uso.

É uma alternativa entre colar tudo no prompt e fazer retrieval do zero o tempo todo.

No espírito do Stanford CS329A, é memória preparada para agentes que aprendem a se organizar.

📊 Comparando arquiteturas de memória

1. Sem memória
Cada turno começa do zero.

Bom para: FAQs simples.

Quebra em: personalização, continuidade e tarefas longas.

2. Curto prazo
Só o histórico da conversa atual.

Bom para: chats curtos.

Problema: esquece entre sessões e sofre com context window.

3. Longo prazo
Vector DB / memória persistente entre sessões.

Bom para: preferências, fatos do usuário e documentos externos.

Problema: ainda depende de retrieval certo e não decide sozinho o que “sai da RAM”.

4. Hierárquica (MemGPT-style)
Contexto ativo + memória externa + política de paginação.

Bom para: agentes persistentes, longos e mais autônomos.

Trade-off: exige orquestração melhor, mas escala muito melhor que “prompt infinito”.

→
Memória persistente
→
Memória hierárquica
🎯 Regra prática para iniciantes: comece com curto prazo + RAG; evolua para memória hierárquica quando o agente precisar durar muito e decidir melhor o que lembrar.

3.6 Skills — capacidades reutilizáveis

📅 Lançado pela Anthropic em outubro/2025.

Conceito: uma "skill" é uma pasta com:

  • Um SKILL.md descrevendo o que faz e quando usar
  • Scripts auxiliares (Python, bash)
  • Templates, dados, recursos

O agente descobre skills disponíveis e carrega só o necessário quando relevante.

~/.claude/skills/
├── pdf-extractor/
│   ├── SKILL.md          ← "use isto para extrair texto de PDFs"
│   └── extract.py
├── jira-integration/
│   ├── SKILL.md
│   └── jira_client.py
└── excel-formatter/
    ├── SKILL.md
    └── format.py
🧠 Por que importa: sem skills, você teria que incluir TODA a documentação de TODAS as ferramentas no system prompt — estourando a janela de contexto e o orçamento. Skills resolvem isso com carregamento preguiçoso.

Skills vs Tools — qual a diferença?

🛠️ Tool
Sempre disponível
Schema JSON fixo
1 função = 1 tool
Ocupa contexto mesmo parada
🎓 Skill
Carrega sob demanda
Pode trazer docs e scripts

🎯 Impacto prático: com 5 tools registradas, ~500 tokens de overhead. Com 50 skills possíveis mas só 3 carregadas, ~300 tokens. Skills escalam melhor para agentes com muitas capacidades.

3.7 MCP — Model Context Protocol

📅 Lançado pela Anthropic em novembro/2024. A documentação oficial define MCP como um padrão open-source para conectar aplicações de IA a sistemas externos.

MCP é o "USB-C" dos agentes:
um protocolo padrão para conectar LLMs a dados, ferramentas e workflows.
Antes do MCP:
Cada framework (LangChain, LlamaIndex, CrewAI…) tinha sua API de tools.
Integrar com Slack? 5 implementações diferentes.
Com MCP:
1 servidor MCP do Slack → funciona em todos os agentes/clientes que falam MCP.
Hoje: Claude, ChatGPT, VS Code, Cursor, Cline, Continue…
🏢 Adoção: além de Claude, o ecossistema inclui ChatGPT, VS Code, Cursor e muitos servidores comunitários. É o "USB-C dos agentes" — conecte uma vez, funcione em muitos clientes MCP.

MCP — arquitetura

Comunicação por JSON-RPC (stdio local ou HTTP/SSE remoto). O host/cliente descobre capacidades expostas por servidores MCP e decide quando usá-las.

MCP — o que ele expõe

📚 Resources
Dados que o agente pode ler: arquivos locais, bases SQL, Notion, Google Drive, calendários, documentação interna.
🛠️ Tools
Ações executáveis: buscar na web, criar issue, consultar API, rodar cálculo, abrir PR, manipular Blender ou Figma.
🧩 Prompts / workflows
Templates especializados e fluxos reutilizáveis: triagem, análise de dados, revisão de código, atendimento, pesquisa.
Modelo mental oficial: MCP não é “mais uma tool”. É a camada de compatibilidade entre clientes de IA e servidores que expõem contexto e ações. O mesmo servidor pode servir muitos clientes.

MCP — o que isso habilita na prática

Assistente pessoal contextual
Conectar Google Calendar + Notion + arquivos para responder “o que preciso preparar para a reunião de amanhã?” com contexto real.
Coding agent com design
Claude Code/Cursor podem ler um design no Figma, editar o repo, rodar testes e propor PR sem integração custom para cada app.
Chatbot enterprise
Um agente consulta múltiplos bancos e sistemas internos com controle de identidade, permissões e auditoria.
Ferramentas criativas
Modelos podem controlar Blender ou pipelines 3D via servidor MCP, transformando linguagem natural em ação em software especializado.
Por que importa: reduz tempo de integração para devs, amplia capacidades dos agentes e dá aos usuários apps que acessam dados e executam ações reais.

3.8 Padrões multi-agente

🚨 Quando um agente não basta: Cursor tentou um único agente para "review + edit + test + deploy". Resultado: contexto estourava, qualidade caía após 10 passos. Solução: supervisor + workers especializados. Cada worker foca em 1 skill com contexto limpo.
1. Supervisor / Orchestrator
Supervisor
→
A / B / C
Um agente central decide qual especialista chamar e agrega os resultados. Os workers não se falam diretamente.
🎯 Quando: tarefas com sub-domínios claros
📦 Ex: customer support (router → billing/tech/sales)
2. Handoff / Swarm
Agente 1
→
Agente 2
→
Agente 3
Cada agente passa o bastão ao próximo quando termina sua parte. Fluxo sequencial, como uma linha de produção.
🎯 Quando: pipeline com etapas ordenadas
📦 Ex: pesquisar → escrever → revisar (OpenAI Swarm)
3. Hierárquico
CEO
→
Managers
→
Workers
Múltiplos níveis: um agente coordena gerentes, que coordenam workers. Escala para tarefas grandes e complexas.
🎯 Quando: projetos grandes, muitos sub-objetivos
📦 Ex: "empresa de agentes" gerando um relatório completo
🔑 Todos compartilham a mesma ideia: dividir para conquistar — cada agente tem contexto e ferramentas próprios, evitando o "agente faz-tudo" que satura.

Quando usar multi-agente?

✅ Use quando
Fases bem diferentes
Prompts/contextos distintos
Especialistas com tools próprias
Paralelismo gera ganho real
❌ Evite quando
Um agente resolve sozinho
Latência é crítica
Custo precisa ser previsível
Você nem mediu o benefício
🎯 Regra prática: single-agent com bom contexto costuma vencer; só vá de multi-agent quando a tarefa realmente pedir especialização ou paralelismo.
🏢 Realidade do mercado (2025): segundo a Anthropic, ~90% dos agentes em produção são single-agent. Multi-agent brilha em: coding assistants (planner + coder + reviewer), research (pesquisador + escritor), e customer support (router + especialistas).

Frameworks multi-agente

Cada framework implementa os padrões anteriores com filosofias diferentes:

👥 CrewAI
"Crew" de agentes com roles (researcher, writer, reviewer). Sintaxe declarativa.
  • Filosofia: modela uma equipe humana — cada agente tem cargo, objetivo e backstory
  • Padrão típico: sequencial (handoff) ou hierárquico
  • Ideal para: protótipos rápidos, fluxos role-based intuitivos
💬 AutoGen (Microsoft)
Agentes conversam entre si em loop até resolver. Forte em código colaborativo.
  • Filosofia: diálogo entre agentes (ex: coder ↔ executor de testes)
  • Padrão típico: conversação em grupo (group chat)
  • Ideal para: geração de código, debate/refinamento iterativo
🔄 LangGraph (multi-agent)
Cada nó do grafo = 1 agente. Controle total sobre o fluxo. Padrão moderno.
  • Filosofia: grafo explícito de estados e transições — você desenha o fluxo
  • Padrão típico: qualquer um (supervisor, handoff, hierárquico)
  • Ideal para: produção, fluxos complexos com loops e condicionais
⚡ OpenAI Agents SDK (ex-Swarm)
Handoffs leves entre agentes. Minimalista, baixo overhead.
  • Filosofia: simplicidade — agentes "transferem" a conversa via function calls
  • Padrão típico: handoff/swarm sequencial
  • Ideal para: roteamento simples, quando você quer pouco "framework"
📌 Como escolher? Protótipo role-based → CrewAI · Código colaborativo → AutoGen · Produção/controle total → LangGraph · Roteamento mínimo → OpenAI Agents SDK

Exemplo: Crew com CrewAI

📖 Em palavras: imagine uma equipe de funcionários. Cada Agent tem um cargo (role), um objetivo e um "currículo" (backstory) — isso vira parte do prompt dele. Cada Task é uma entrega esperada, atribuída a um agente. O Crew orquestra: o pesquisador entrega, o escritor pega o resultado dele como contexto e produz o artigo. Divisão de trabalho automatizada. 
from crewai import Agent, Task, Crew
pesquisador = Agent(
    role="Pesquisador",
    goal="Encontrar informações precisas sobre o tema",
    backstory="Você é um analista meticuloso.",
    tools=[busca_web, leitor_pdf],
)
escritor = Agent(
    role="Escritor técnico",
    goal="Transformar pesquisa em artigo claro",
    backstory="Você escreve para devs juniores.",
)

Exemplo: Crew com CrewAI — continuação

t1 = Task(description="Pesquise tudo sobre {tema}", agent=pesquisador)
t2 = Task(description="Escreva artigo de 500 palavras baseado na pesquisa",
          agent=escritor, context=[t1])
crew = Crew(agents=[pesquisador, escritor], tasks=[t1, t2])
print(crew.kickoff(inputs={"tema": "agentes de IA"}))

3.9 Hands-on integrado — agente com RAG + memória

📋 Colab-ready — usa HuggingFace InferenceClient (token gratuito)
# 📋 Execute no Google Colab — requer client do setup
from typing import TypedDict

# --- Base de conhecimento local (simula vector store) ---
KNOWLEDGE_BASE = [
    "RAG combina retrieval com geração para reduzir alucinações.",
    "Memória persistente permite continuidade entre sessões.",
    "Vector stores indexam documentos por similaridade semântica.",
    "ChromaDB é um vector store open-source em Python.",
]

def retrieve(query: str) -> list[str]:
    """Busca por keyword (em produção: embeddings + similaridade)."""
    return [d for d in KNOWLEDGE_BASE if any(w in d.lower() for w in query.lower().split())] or KNOWLEDGE_BASE[:2]

def generate(context: list[str], question: str) -> str:
    """Gera resposta usando HuggingFace com contexto injetado."""
    ctx = "\n".join(f"- {d}" for d in context)
    resp = client.chat.completions.create(
        model="Qwen/Qwen2.5-7B-Instruct",
        messages=[
            {"role": "system", "content": f"Responda baseado APENAS neste contexto:\n{ctx}"},
            {"role": "user", "content": question},
        ],
        max_tokens=200,
    )
    return resp.choices[0].message.content

class State(TypedDict):
    messages: list[dict]
    context_docs: list[str]

def retrieve_node(state: State) -> dict:
    q = state["messages"][-1]["content"]
    docs = retrieve(q)
    print(f"🔍 Recuperados {len(docs)} docs para: {q[:40]}")
    return {"context_docs": docs}

3.9 Hands-on integrado — continuação

def agent_node(state: State) -> dict:
    resposta = generate(state["context_docs"], state["messages"][-1]["content"])
    return {"messages": [{"role": "assistant", "content": resposta}]}

# --- Grafo simples: retrieve → generate ---
def run_rag_agent(pergunta: str) -> str:
    state: State = {"messages": [{"role": "user", "content": pergunta}], "context_docs": []}
    state.update(retrieve_node(state))
    state.update(agent_node(state))
    return state["messages"][-1]["content"]

# Teste — chamadas reais ao HuggingFace!
print(run_rag_agent("O que é RAG e como funciona?"))
print(run_rag_agent("Como funciona memória persistente?"))
💾 Padrão: retrieve → generate. Em produção, troque retrieve() por ChromaDB + embeddings. A estrutura do código permanece idêntica — só muda o retriever.
