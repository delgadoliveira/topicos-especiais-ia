💸 Context window = budget de dados

❌ Intuição errada

“Joga tudo no prompt.”

  • janela é finita
  • latência sobe
  • custo por requisição explode
  • risco de expor dado demais

✅ Visão correta

Prompt é orçamento.

  • contexto deve ser mínimo
  • cada token precisa justificar valor
  • acesso segue política
  • menos contexto irrelevante = mais precisão

📦 Três estratégias para “dar dados” ao agente

Context stuffing
cola contexto fixo no prompt
barato, rápido, mas limitado
RAG
busca só o que importa
escalável e atualizável
Fine-tuning
muda comportamento do modelo
caro e mais permanente
Analogia: stuffing = levar uma pasta; RAG = consultar o arquivo certo; fine-tuning = treinar um novo funcionário.

📊 Decisão de negócio: qual estratégia usar?

Para dados corporativos vivos, o default executivo costuma ser: RAG + governança.

🔍 RAG para executivos

Retrieval-Augmented Generation = buscar pedaços relevantes de documentos e injetar só esses trechos no prompt.
Valor: conhecimento atualizado sem retreinar o modelo.
Risco: se a base está ruim, o agente erra com muita confiança.
Analogia: o agente não “decorou” a biblioteca; ele chama um bibliotecário antes de responder.

⚙️ Pipeline simplificado de RAG

documento bruto → chunking
chunk → embedding
embedding → vector DB
pergunta do usuário → busca semântica + filtros
top-k trechos → prompt final
modelo responde com base nos trechos recuperados
Cada etapa é uma superfície de custo, qualidade e governança.

🏦 Onde isso gera valor real

Banco digital
FAQ interno atualizado
menos tickets repetitivos
Jurídico
busca em contratos e cláusulas
menos tempo por revisão
Compliance
normas, políticas, evidências
mais rastreabilidade
Business value: manter a base viva custa menos do que reeducar o modelo toda semana.

🗑️ Limitações honestas do RAG

Problemas comuns
  • documento desatualizado
  • chunk mal cortado
  • metadata inconsistente
  • busca sem filtros
Mitigação
  • curadoria da base
  • reindexação versionada
  • testes de retrieval
  • logs de acesso por consulta
Analogia: um buscador sobre arquivos mal catalogados só digitaliza a bagunça do armário.

🔐 O problema central de governança

Sem controle de acesso, o melhor RAG do mundo vira o melhor mecanismo de vazamento da empresa.
Todo usuário
vê todo chunk
Toda pergunta
consulta toda base
Toda resposta
pode revelar demais
Governança em IA não começa na resposta. Começa antes da busca.

🏷️ Metadata filtering

Cada chunk recebe tags
{departamento: "RH", nivel: "gerente+", pais: "BR"}
Na consulta
o sistema filtra permissões antes da busca semântica final.
Analogia: não basta achar o documento certo; é preciso confirmar se a pessoa pode entrar na sala onde ele está.

🛂 RBAC aplicado ao RAG

user: “Mostre a política de bônus do board”
perfil: analista_junior / Finanças / BR
policy_engine() → permitido? não
vector_search() → consulta bloqueada
assistant: acesso insuficiente; contate RH executivo
A negativa também é produto. Ela precisa ser clara, auditável e educada.

📁 Case: 50 mil documentos, 3 níveis, 5 departamentos

Arquitetura mínima
  • chunks com owner e sensibilidade
  • índices por domínio
  • policy engine central
  • logs por consulta e resposta
Benefício executivo
  • menos risco de vazamento lateral
  • mais confiança para escalar uso
  • compliance mais fácil de provar
  • custos previsíveis por área

🧭 Quando usar RAG vs fine-tuning vs prompt engineering

Dados mudam sempre? → RAG
Quer mudar estilo, formato ou comportamento persistente? → Fine-tuning
São poucas regras e poucos exemplos? → Prompt engineering
PM rule: não use fine-tuning para resolver um problema que é, na verdade, de acesso a dados.

🧮 Custo mensal de manutenção — visão simplificada

Fine-tunealtoaltomudança frequente
Analogia financeira: fine-tuning é CAPEX alto; RAG é OPEX controlável; stuffing é caixa pequeno com teto baixo.

🔌 MCP como ativo estratégico

Model Context Protocol é um padrão para conectar agentes a sistemas como ERP, CRM, bancos e APIs internas.
Antes: integrações customizadas para cada combinação de agente e sistema.
Depois: um servidor padronizado por sistema, reutilizável por vários agentes.
Analogia: USB-C para IA. Menos adaptadores, menos código artesanal, menos risco operacional.

📉 Antes do MCP vs com MCP

❌ Sem padrão

N agentes × M sistemas

  • muito código duplicado
  • segurança inconsistente
  • onboarding lento
  • troca de fornecedor cara

✅ Com MCP

N agentes + M servidores

  • contrato único
  • telemetria padronizada
  • menos acoplamento
  • reuso entre produtos

💼 ROI executivo do MCP

60–80%
menos código custom de integração
Mais velocidade
para plugar novos agentes
Mais governança
logs e políticas centralizados
Exemplos: servidor MCP para SAP, outro para Salesforce, outro para Postgres.
Ecossistema já adotado por Claude, Cursor, GitHub Copilot e Windsurf.

1️⃣ Step 1 — Base local com ChromaDB

# Documentos mock
DOCS = [
    ("d1", "Política de alocação da carteira conservadora.", "publico"),
    ("d2", "Estratégia interna para rebalanceamento trimestral.", "interno"),
    ("d3", "Comitê aprovou limite setorial de 12% em energia.", "restrito"),
]

client = chromadb.Client()
col = client.create_collection("investimentos")

for doc_id, texto, nivel in DOCS:
    col.add(ids=[doc_id], documents=[texto], metadatas=[{"nivel": nivel}])

2️⃣ Step 2 — Metadata filtering

def buscar_documentos(pergunta, papel):
    niveis = list(PERMISSOES[papel])
    return col.query(
        query_texts=[pergunta],
        n_results=2,
        where={"nivel": {"$in": niveis}},
    )

print(buscar_documentos("limite de energia", "senior"))
# senior não deve ver conteúdo "restrito"
Analogia: a busca já sai do elevador no andar permitido; ela nem passa na porta do board.

3️⃣ Step 3 — Router por intenção

def classificar_rota(pergunta):
    q = pergunta.lower()
    if any(x in q for x in ["agora", "hoje", "tempo real", "cotação"]):
        return "api_mercado"
    if any(x in q for x in ["política", "comitê", "rebalanceamento"]):
        return "rag_local"
    return "llm_direto"

print(classificar_rota("Qual a cotação do dólar agora?"))
PM view: roteamento é política operacional, não truque técnico.

4️⃣ Step 4 — Resolver pergunta com custo

def responder(pergunta, papel="junior"):
    rota = classificar_rota(pergunta)
    custo = CUSTOS[rota]
    if rota == "rag_local":
        dados = buscar_documentos(pergunta, papel)["documents"][0]
        return {"rota": rota, "custo": custo, "resposta": dados}
    if rota == "api_mercado":
        return {"rota": rota, "custo": custo, "resposta": "Cotação simulada: 5.42"}
    return {"rota": rota, "custo": custo, "resposta": "Resposta geral do assistente"}

5️⃣ Step 5 — Simular 100 queries

queries = [
    "Explique diversificação", "Qual a cotação do dólar agora?",
    "Mostre a política de alocação", "Qual o outlook do setor elétrico?",
] * 25

rotas = [classificar_rota(q) for q in queries]
contagem = Counter(rotas)

custo_total = sum(CUSTOS[r] for r in rotas)
print(contagem)
print(f"Custo total: ${custo_total:.3f}")
