🤖 IA (Inteligência Artificial) — software que faz coisas que normalmente exigiriam pessoas

Definição formal: Campo da ciência da computação dedicado a criar sistemas que realizam tarefas que requerem inteligência humana.

Na prática hoje: IA não é uma coisa só — é um guarda-chuva. Inclui desde o filtro de spam do seu email (ML clássico dos anos 2000) até o ChatGPT (LLM generativo de 2022).

Tipos que você vai encontrar:

  • IA Narrow/Fraca — resolve UMA tarefa (ex: detectar fraude)
  • IA Generativa — cria conteúdo novo (texto, imagem, código)
  • AGI (General) — resolve QUALQUER tarefa como humano (ainda não existe)

🎯 Analogia: "IA" é como "veículo" — pode ser bicicleta, carro, ou foguete. Quando alguém fala "IA", pergunte: que tipo?

🧠 LLM (Large Language Model) — o "cérebro" por trás do ChatGPT, Claude, Gemini

O que é: Uma rede neural com bilhões de parâmetros, treinada em trilhões de palavras da internet, livros e código. Seu único objetivo durante o treino: prever a próxima palavra.

Por que funciona: Para prever bem a próxima palavra, o modelo precisa "entender" gramática, lógica, fatos, humor, código — tudo. É como um estudante que, de tanto ler, desenvolve raciocínio.

Escala importa:

  • GPT-2 (2019): 1.5 bilhão de parâmetros → texto OK
  • GPT-3 (2020): 175 bilhões → emergent abilities aparecem
  • GPT-4 (2023): ~1 trilhão (estimado) → raciocínio avançado

🎯 Analogia: Um LLM é como alguém que leu TODOS os livros da biblioteca — sabe muito, mas não pode sair da biblioteca pra verificar nada. Agentes dão "pernas" a esse cérebro.

💬 Prompt — o texto/instrução que você envia para o LLM

Simples assim: O prompt é a sua mensagem. Pode ser uma pergunta, um comando, ou um contexto inteiro com exemplos.

Por que importa tanto: O mesmo modelo dá respostas radicalmente diferentes dependendo de COMO você pergunta. Prompt engineering é a arte de comunicar bem com IAs.

Tipos de prompt:

  • Zero-shot: "Traduza: Hello → " (sem exemplos)
  • Few-shot: "Hello→Olá, Good→Bom, World→" (com exemplos)
  • System prompt: Define a personalidade/regras do assistente
  • Chain-of-Thought: "Pense passo a passo antes de responder"

🎯 Analogia: O prompt é como o briefing que você dá a um estagiário brilhante mas literal. Quanto mais claro e específico, melhor o resultado.

🔌 API (Application Programming Interface) — a "tomada" para conectar seu código a serviços externos

Em termos simples: API é um contrato: "me mande X neste formato, e eu te devolvo Y". É como um cardápio de restaurante — você pede, a cozinha entrega.

No contexto de IA: Em vez de abrir o site do ChatGPT e digitar, seu código Python manda uma requisição HTTP para a API da OpenAI e recebe a resposta em JSON.

Por que agentes precisam de APIs:

  • Chamar o LLM programaticamente (OpenAI, Anthropic, Google)
  • Acessar ferramentas (clima, busca, banco de dados, email)
  • Conectar sistemas entre si sem intervenção humana

Custo: APIs de LLM cobram por token (~$0.01–$0.06 por 1K tokens de saída). Um agente pode fazer 5–50 chamadas por tarefa.

🎯 Analogia: Se o LLM é o cérebro e as ferramentas são as mãos, a API é o sistema nervoso que conecta tudo.

🦾 Agente de IA — um LLM com permissão e capacidade de AGIR no mundo

A diferença crucial: Um chatbot CONVERSA. Um agente EXECUTA. O agente decide sozinho o que fazer, faz, observa o resultado, e decide o próximo passo.

Componentes de um agente:

  • LLM (cérebro) — raciocina e decide
  • Ferramentas — busca web, calculadora, APIs, banco de dados
  • Loop — pensa → age → observa → repete até resolver
  • Memória — lembra do contexto e de interações passadas

Exemplos reais agora:

  • Cursor/Copilot: lê seu código, edita, roda testes, corrige erros
  • Perplexity: busca na web, lê sites, sintetiza resposta com fontes
  • Customer service bots: consulta pedido, processa reembolso, atualiza sistema

🎯 Analogia: ChatGPT = Google Maps que te mostra o caminho. Agente = motorista do Uber que te LEVA até lá.

🎬 Uma cena para fixar a ideia

Imagine que você pede ao ChatGPT:

"Qual foi o time campeão do Brasileirão 2024 e quantos gols o artilheiro fez?"
🚫 LLM puro (sem ser agente)
"Não tenho informações sobre eventos posteriores a janeiro de 2024..."
Ou pior: inventa uma resposta plausível mas errada (alucinação).

Por que ele não sabe? O LLM é um arquivo estático — uma "foto" do conhecimento do mundo no momento em que foi treinado. Depois do treino, ele nunca mais aprende nada novo.

O processo:

  • Treino (meses, milhões de $): lê bilhões de textos até uma data de corte (ex: jan/2024)
  • Deploy: o modelo "congela" — os pesos nunca mais mudam
  • Inferência: quando você pergunta algo, ele combina estatisticamente o que já sabe

Consequências práticas:

  • ❌ Não sabe o resultado do Brasileirão 2024 (aconteceu depois do corte)
  • ❌ Não sabe o preço do dólar HOJE (muda a cada segundo)
  • ❌ Não pode verificar se uma informação ainda é verdade

E a "alucinação"? Quando não sabe, o LLM não diz "não sei" — ele gera texto que parece correto porque é estatisticamente plausível. Ex: pode dizer "Palmeiras foi campeão" com confiança total, simplesmente porque Palmeiras é frequente em contextos de título.

⚠️ Regra de ouro: LLM sem ferramentas = enciclopédia desatualizada que nunca admite ignorância. Por isso precisamos de agentes.
✅ Agente
1. Percebe que precisa de dado atualizado
2. Chama uma ferramenta de busca na web
3. Lê os resultados
4. Verifica e responde: "Botafogo foi campeão; Alerrandro foi artilheiro com 19 gols. Fonte: …"
Essa é a diferença prática entre LLM e agente: o agente sabe quando pedir ajuda e sabe como usar ferramentas.

1.2 O que é um Agente de IA?

Existem dezenas de definições. A mais útil, pragmática, vem da Anthropic (2024):

Um agente é um sistema onde um LLM dirige seu próprio fluxo de execução em loop,
decidindo quais ferramentas chamar, com base nas observações que recebe,
até atingir um objetivo.
Vamos quebrar essa definição em partes nos próximos slides.

🤖 Agente vs Assistente vs Bot

Mesmo pedido — três respostas completamente diferentes:

🗣️ Usuário: "Preciso de um voo barato pra São Paulo na sexta"
🔧 Bot
"Digite 1 para consultar voos
Digite 2 para pacotes
Digite 3 para falar com atendente"

O que é: Fluxo pré-programado (if/else). Não entende linguagem natural.

Problema: Se você disser "quero o mais barato pra sexta", ele responde "opção inválida".

Autonomia: Zero — só segue o script que o programador escreveu.

Ex: URA telefônica, bots de WhatsApp com menu numérico, chatbots de FAQ com botões.
💡 Assistente
"Para encontrar voos baratos, recomendo:
1. Acesse Google Flights
2. Coloque SP como destino
3. Selecione sexta-feira..."

O que é: LLM que entende sua pergunta e responde com informação útil.

Problema: Te diz como fazer, mas não faz por você. Você ainda precisa agir.

Autonomia: Baixa — responde e sugere, mas não executa nenhuma ação.

Ex: ChatGPT padrão, Alexa respondendo perguntas, Google Assistant em modo informativo.
🦾 Agente
"Encontrei 3 opções. A mais barata é LATAM
às 7h por R$289. Reservo no seu cartão
final 4421? ✈️"

O que é: LLM + ferramentas + loop. Ele pesquisou, comparou, decidiu e está pronto pra agir.

Diferencial: Tomou decisões autônomas e pode executar a ação (reservar) se autorizado.

Autonomia: Alta — planeja, executa, verifica resultado, adapta se algo der errado.

Ex: Copilot Workspace, agentes de viagem, Cursor editando código, Devin deployando apps.
Script fixo → Entende + sugere → Entende + decide + executa

Tipos de Agentes

Agentes não são todos iguais — existem diferentes dimensões de classificação:

👁️ Por interação com o usuário
Surface agents — conversam, explicam, pedem confirmação
Background agents — trabalham sozinhos, entregam resultado pronto

Surface agent (interativo): Chatbot do Nubank — conversa com o cliente, pede CPF, explica o que vai fazer, espera confirmação antes de agir. O humano vê cada passo.

Background agent (autônomo): GitHub Copilot Workspace — você abre uma issue, o agente planeja, implementa, testa e abre o PR. Você só vê o resultado final.

Qual usar? Surface para decisões sensíveis (financeiro, saúde). Background para tarefas repetitivas onde o resultado é verificável (CI/CD, data pipelines, código).

Cursor = surface (mostra diffs, pede approve). Devin = mais background (trabalha sozinho em branches).
👥 Por quantidade de agentes
Single-agent — um cérebro, todas as ferramentas
Multi-agent — especialistas que colaboram

Single-agent: Um agente com todas as ferramentas (busca, código, email). Simples de debugar, mas fica sobrecarregado em tarefas complexas.

Multi-agent: Pesquisador + Programador + Revisor, cada um com seu prompt e ferramentas. Coordenados por um "orquestrador".

Quando usar multi-agent? Quando a tarefa envolve papéis muito diferentes (ex: gerar código E testar E fazer deploy) ou quando especialização melhora qualidade.

CrewAI, AutoGen e LangGraph suportam multi-agent. Vamos praticar no Encontro 3.
💬 Por formato de ação
JSON Agent — saída estruturada em JSON
Code Agent — gera código executável
Function-calling — protocolo nativo do modelo

JSON Agent: O LLM gera {"action": "search", "input": "clima SP"}. Um parser lê e executa. Simples, mas limitado em lógica complexa.

Code Agent: O LLM gera Python completo com loops, condicionais, imports. Mais expressivo — pode fazer cálculos, processar dados, compor lógica. Usado pelo smolagents (HuggingFace).

Function-calling: O modelo foi treinado pra emitir chamadas de função no formato da API (OpenAI, Anthropic). O runtime intercepta e executa. É o mais confiável em produção.

OpenAI usa function-calling. smolagents usa Code Agent. LangChain suporta ambos.
⭐ Por nível de agência
Do processador simples ao multi-agent autônomo
(Espectro HuggingFace/smolagents)

1. 🎭 Persona + Objetivo (System Prompt) — foco e consistência

O que é: instruções que definem quem o agente é, o que deve fazer e quais limites respeitar.

Sem ele: o LLM é genérico — responde qualquer coisa, sem identidade nem limites.

Com ele: se comporta como um especialista com missão clara. Ex: "Você é um auditor que NUNCA aprova despesas sem justificativa."

Analogia: é a "descrição de cargo" de um funcionário.

2. 🧠 Modelo / Reasoning Engine (LLM) — raciocínio e decisão

O que é: o motor de raciocínio — GPT-4, Claude, Gemini, Llama...

Capacidade: interpreta contexto, decompõe problemas, planeja e decide qual ação tomar.

Limitação: sozinho, ele só gera texto. Não acessa internet, não faz cálculos precisos, não lembra conversas.

Analogia: um consultor brilhante trancado numa sala — sem telefone, sem computador e sem arquivo.

3. 💾 Memória — continuidade e contexto

O que é: sistema que armazena e recupera informações entre turnos e sessões.

Curto prazo: histórico da conversa atual. Longo prazo: vector DB com docs e preferências.

Sem memória: cada interação começa do zero.

Com memória: "Na semana passada você disse que prefere hotéis centrais. Mantendo."

Analogia: médico que vê seu prontuário vs um que nunca te viu.

4. 🛠️ Tools + Executor — ação no mundo real

O que é: funções externas (APIs, DB, browser, calculadora) que o agente invoca.

Sem tools: inventa respostas e não faz nada concreto.

Com tools: consulta cotação real, envia email, cria arquivo, reserva hotel.

O executor chama a função e devolve o resultado para o LLM continuar.

Analogia: dar mãos, telefone e internet ao consultor trancado na sala.

5. 🔄 Loop de Controle — autonomia e iteração

O que é: o ciclo Observe → Think → Act → Reflect que se repete até concluir.

Sem loop: chatbot — 1 pergunta, 1 resposta, fim.

Com loop: tenta, falha, ajusta e tenta de novo. 10+ passos autônomos.

Inclui: condição de parada, limite de iterações, detecção de loops infinitos.

Analogia: "escreva um artigo" vs "pesquise, escreva, revise, corrija e publique".

Ciclo de vida de UM turno do agente

O "loop" parece simples, mas internamente cada iteração tem 7 etapas:

1. Input
→
2. Contexto
→
3. Chamada LLM
→
4. Saída
→
5. Validação
→
6. Execução
→
7. Observação
1. Input: pergunta do usuário ou observação do passo anterior chega ao sistema.
2. Contexto: monta o "prompt completo" — system + histórico + tools schema + input. Aqui se decide o que cabe na janela.
3. Chamada LLM: envia tudo à API. O modelo processa e gera uma resposta (texto ou tool_call).
4. Saída: o modelo retorna: "quero chamar a tool X com args Y" ou "aqui está a resposta final".
5. Validação: o sistema verifica: a tool existe? Os args estão no formato certo? Há permissão?
6. Execução: se validou, roda a função real (API call, DB query, cálculo). Se não, retorna erro ao LLM.
7. Observação: resultado da tool é injetado no histórico como "observation" → volta ao passo 2.
💰 Cada iteração custa uma chamada à API. Com 10 passos e contexto crescente, o custo pode multiplicar 10×. Agentes típicos rodam de 3 a 30 iterações por tarefa.

🤖 Mas afinal, o que é um LLM?

LLM = Large Language Model (Modelo de Linguagem de Grande Escala)
Definição simples:
Um programa de computador que aprendeu a prever a próxima palavra lendo bilhões de textos da internet. Ele não "entende" como humanos — mas identifica padrões estatísticos tão sofisticados que parecem compreensão.
🧩 Analogia:
Imagine alguém que leu toda a Wikipédia, Stack Overflow, livros e Reddit. Não memorizou frases — mas aprendeu padrões de como texto funciona. Quando você pergunta algo, ele gera a continuação mais "plausível" baseada nesses padrões.
📊 Escala:
GPT-4 leu ~13 trilhões de tokens no pré-treino. É como ler 10 milhões de livros.
💡 Emergência:
Capacidades como raciocínio, humor e código surgiram "espontaneamente" com escala — ninguém as programou explicitamente.
⚠️ Limite:
Conhecimento congelado na data do treino. Não acessa internet, não atualiza sozinho, pode inventar fatos com confiança.
⚠️ LLM ≠ base de dados. Ele não "guarda" frases; ele aprendeu estatísticas de como palavras se relacionam. Por isso pode inventar (alucinar) — ele gera o que é estatisticamente plausível, não necessariamente verdadeiro.
Exemplos: GPT-4o, Claude Sonnet, Gemini 2.5 Pro, Llama 4, Qwen 3 — todos são LLMs com arquiteturas similares (Transformer).
💡 Analogia simples: Imagine um autocomplete do celular, mas treinado com toda a internet. Ele não "entende" — ele prevê o que vem depois com altíssima precisão. Tão boa que parece inteligência.

🏗️ Como um LLM é treinado?

O treinamento tem 3 fases — cada uma adiciona uma camada de capacidade diferente.
📚 Pré-treino
→
🎯 Fine-tuning
→
👍 RLHF
1. Pré-treino (meses, milhões de USD)

O modelo lê trilhões de tokens da web: livros, código, Wikipedia, fóruns, papers, notícias.

Objetivo: prever o próximo token. Esse objetivo simples faz o modelo aprender gramática, fatos, lógica e até raciocínio.

Resultado: um "fundation model" — sabe muito, mas não segue instruções (é um "autocomplete" gigante).

Custo: GPT-4 custou ~$100M em compute. Llama 3.1 usou 16.000 GPUs por meses.

2. Fine-tuning supervisionado (SFT)

Treina o modelo em exemplos curados de "pergunta → resposta ideal", escritos por humanos.

Objetivo: ensinar o formato de diálogo — como seguir instruções, responder perguntas, recusar pedidos perigosos.

Resultado: o modelo agora conversa. Antes: completava texto. Agora: responde perguntas.

Escala: tipicamente 10K-100K exemplos curados (muito menos que pré-treino).

3. RLHF (Reinforcement Learning from Human Feedback)

Humanos comparam pares de respostas e escolhem a melhor. Um "reward model" aprende esse ranking.

Objetivo: alinhar o modelo com preferências humanas — respostas mais úteis, seguras, honestas e naturais.

Resultado: o modelo melhora em nuance: não é só correto, é agradável de usar.

Variantes modernas: DPO (Direct Preference Optimization), RLAIF (feedback de outro AI), Constitutional AI.

🎯 Para agentes: o fine-tuning e RLHF são onde modelos aprendem a usar tools, seguir formatos JSON e "pensar antes de agir" (CoT). Modelos agentic são treinados com exemplos de tool use.
🎯 Por que isso importa para agentes? O treinamento define os limites do que o agente pode fazer. Um modelo treinado até 2024 não sabe o que aconteceu em 2025 — por isso precisa de ferramentas (busca web) para informação atual.

🧱 A arquitetura: Transformer (simplificado)

Desde 2017 ("Attention is All You Need" — Google Brain), todos os LLMs usam a mesma base: a arquitetura Transformer.
Texto
→
Tokenização
→
Embeddings
→
Self-Attention
→
Feed-Forward
→
Próximo token
1. Tokenização — texto vira números

O texto é partido em pedaços (tokens). "Inteligência" → ["Int", "elig", "ência"]. Cada token vira um ID numérico (ex: 15496).

Por que não usar palavras inteiras? Vocabulário finito. Tokens permitem lidar com qualquer idioma, neologismo ou código.

2. Embeddings — números viram vetores com significado

Cada token ID é mapeado para um vetor de alta dimensão (ex: 4096 números). Palavras similares ficam próximas nesse espaço.

Exemplo: "rei" - "homem" + "mulher" ≈ "rainha". O modelo aprendeu relações semânticas como geometria.

3. Self-Attention — cada palavra olha para todas as outras

O mecanismo que tornou Transformers revolucionários. Cada token calcula "quanto atenção devo dar" a cada outro token na sequência.

Exemplo: em "O gato que eu vi ontem dormia", "dormia" presta muita atenção a "gato" (o sujeito) e ignora "eu" e "ontem".

Multi-head: vários "cabeças" de atenção em paralelo capturam relações diferentes (sintaxe, semântica, coreference).

4. Feed-Forward — processamento profundo

Após a atenção, cada token passa por redes neurais densas que transformam a representação. É aqui que "conhecimento factual" é armazenado nos pesos.

Esse bloco (Attention + FFN) é repetido dezenas/centenas de vezes em camadas. Mais camadas = raciocínio mais profundo.

📐 Escala importa
GPT-2: 12 camadas, 1.5B parâmetros
GPT-4: ~120 camadas, ~1.8T parâmetros
Llama 4 Maverick: 128 experts, 400B ativo
🧩 Analogia
Cada camada é como uma "rodada de discussão" num time. 12 rodadas = opinião superficial. 120 rodadas = análise profunda com múltiplas perspectivas.

🎯 O que o LLM realmente faz: prever o próximo token

Essa é a ideia mais importante para entender LLMs:
O LLM faz UMA coisa: dado um texto de entrada, calcula a probabilidade de cada possível próximo token e escolhe um. 
Entrada: "O céu é"  →  Saída mais provável: "azul" (87%), "bonito" (5%), "claro" (3%)
Cada cor = 1 token. Aproximação BPE simplificada (~GPT-4 tokenizer).
🎯 Por que predição = inteligência?
Para prever bem em bilhões de contextos, precisa aprender gramática, lógica, fatos, código... Tudo "emerge" do objetivo de predição.
⚠️ Consequência perigosa:
Gera o que é estatisticamente plausível, não o que é verdadeiro. Daí vêm as alucinações.

Capacidades vs Limitações — e por que precisamos de agentes

Esse mecanismo de predição é poderoso — mas tem limites fundamentais que agentes resolvem.
✅ O que ele faz bem:
• Completar padrões conhecidos
• Seguir instruções complexas
• Raciocinar sobre texto (multi-step)
• Gerar código, resumos, traduções
• Adaptar tom, estilo, formato
• Sintetizar informação de contexto
❌ O que ele NÃO faz (sozinho):
• Acessar a internet em tempo real
• Lembrar conversas anteriores
• Executar código ou ações
• Garantir fatos (pode alucinar)
• Atualizar seu conhecimento
• Interagir com sistemas externos
💡 E é exatamente por isso que precisamos de AGENTES — para dar ao LLM: mãos (tools), memória (persistence), olhos (retrieval) e autonomia (loop de controle). Cada limitação vira uma oportunidade de composição.
📊 Em números: ChatGPT padrão resolve ~60% de tarefas complexas. Com agente + tools + RAG, o mesmo modelo sobe para ~85-95% (benchmark GAIA, 2024).

Se o agente ficou caro, lento ou inconsistente → quase sempre são 3 suspeitos: tokens · janela de contexto · temperatura. Começando pelo primeiro:

🌡️ Temperature (Temperatura)

A temperature controla o quão previsível vs criativo o modelo será ao escolher o próximo token.
🧊
temp = 0
Mais provável, repetível.
Ideal para agentes e código
😊
temp = 0.7
Mais variedade, ainda coerente.
Ideal para chat e escrita
🎲
temp = 1.5+
Muito aleatório; pode gerar nonsense.
Ideal para brainstorm e arte

temp=0.7: "sofá", "tapete", "telhado".
temp=1.5: até "existencialismo".

🎨 Temperatura 0.7-1.0 (criativo)
Ideal para: brainstorming, geração de texto criativo, exploração de ideias
🧩 Analogia: responder prova em temp alta é "chutar com estilo". Às vezes brilha; geralmente erra mais.

🛠️ Anatomia de uma Tool — 3 partes obrigatórias

Para o LLM saber quando e como chamar uma ferramenta, ele precisa de um "contrato" com 3 peças:

① Nome + Descrição
O LLM escolhe a tool pela descrição. Se for vaga, ele chama errado.
{
  "name": "search_web",
  "description": "Busca informações
    atuais na internet sobre
    qualquer tema."
}
💡 Pense na descrição como um anúncio para o LLM.
② Schema dos Parâmetros
JSON Schema define tipos, obrigatórios e valores possíveis.
{
  "query": {
    "type": "string",
    "description": "O que buscar"
  },
  "max_results": {
    "type": "integer",
    "default": 5
  }
}
💡 Sem schema = o LLM inventa argumentos.
③ Implementação (runtime)
O LLM nunca executa — só pede. Quem roda é o seu código.
def search_web(query, max_results=5):
    results = requests.get(
        f"https://api.search/{query}"
    )
    return results.json()[:max_results]
💡 A tool pode ser uma API, um script, um SQL…
⚡ Analogia: É como um cardápio de restaurante. O nome = nome do prato, a descrição = ingredientes, o schema = como personalizar (ponto da carne, acompanhamento). O chef (runtime) é quem cozinha — o cliente (LLM) só pede.

Porque agir é diferente de falar.
Sem tools bem descritas, ele vira um ótimo narrador de intenções.

3 categorias de tools:
📖 Retrieval (leitura)

Busca informação que o LLM não tem. Ex: busca web, consulta a banco de dados, RAG sobre documentos internos, consulta de FAQ.

Risco: baixo — apenas lê dados. Atenção à exposição de dados sensíveis.

⚡ Computação / I/O

Executa cálculos ou manipula arquivos. Ex: rodar Python, gerar gráfico, converter PDF, fazer query SQL de leitura.

Risco: médio — pode consumir recursos ou acessar sistema de arquivos.

🚀 Ação (escrita)

Modifica o mundo real. Ex: enviar email, criar PR no GitHub, fazer deploy, deletar registro, transferir dinheiro.

Risco: alto — ações irreversíveis. Requer confirmação humana ou guardrails rigorosos.

⚠️ Princípio de mínimo privilégio:
Dê ao agente só as tools que ele precisa. Leitor de email ≠ enviador. Query SQL ≠ DELETE. Cada tool extra é superfície de ataque.

OpenAI Function Calling · Anthropic MCP · Google Gemini Tools
Catálogos: Composio, Arcade, Zapier (1000+ integrações prontas)

💾 Deep dive: memória — 3 camadas

Sem memória, cada turno é amnésico. Com memória, o agente aprende, personaliza e acumula contexto entre sessões.
⚡ Curto prazo
↔
📝 Working memory
↔
🗄️ Longo prazo
⚡ Curto prazo (in-context)

O que é: as mensagens da conversa atual que vão no prompt. Tudo que está na "janela de contexto".

Limite: quando enche, as mensagens mais antigas são descartadas (sliding window) ou resumidas.

Exemplo: ChatGPT lembra o que você disse 10 mensagens atrás — mas esquece se a conversa for longa demais.

Custo: cresce linearmente. 128K tokens de contexto = ~$2-6 por chamada com GPT-4.

📝 Working memory (scratchpad)

O que é: espaço estruturado que o agente usa para anotar decisões intermediárias sem inflar o histórico inteiro.

Exemplo: "Plano atual: 1) buscar dados ✓ 2) calcular média → em andamento 3) formatar relatório"

Implementação: variável separada no prompt, atualizada a cada turno. Frameworks como LangGraph usam state.

Analogia: post-its na mesa enquanto trabalha — não vai pro arquivo permanente.

🗄️ Longo prazo (persistente)

O que é: banco de dados externo (vector DB, SQL, grafos) que persiste entre sessões.

3 subtipos:

• Episódica: o que aconteceu ("usuário cancelou pedido em 12/jan")

• Semântica: fatos ("política de reembolso: até 30 dias")

• Procedural: como fazer ("para cancelar, chame API X com param Y")

Tecnologias: Pinecone, Weaviate, Qdrant, pgvector, ChromaDB.

🏢 Mercado: ChatGPT Memory (longo prazo automático) · Claude Projects (docs como contexto) · Cursor @-context (codebase inteira) · Mem0 e Letta/MemGPT (memória gerenciada por IA)
⚠️ Problema real: memória longa sem curadoria vira "lixo acumulado". O agente precisa de estratégias de esquecimento (TTL, relevância) tanto quanto de lembrar.
🏢 Exemplo real: O Copilot (GitHub) usa memória de curto prazo (seu arquivo aberto), working memory (arquivos recentes do projeto), e longo prazo (convenções do repositório) para sugerir código contextualizado.

🔄 Deep dive: o loop de controle

O loop é o que transforma um chatbot (1 pergunta → 1 resposta) em um agente autônomo (objetivo → N passos → resultado).
📥 Input
→
🧠 Think
→
🛠️ Act
→
👁️ Observe
↩
🧠 Think
O que acontece em cada iteração

1. Think: LLM analisa o contexto e decide: "já tenho a resposta final?" ou "preciso de mais uma ação?"

2. Act: Se precisa agir, gera um tool_call com nome + argumentos em JSON.

3. Observe: Sistema executa a tool, captura o resultado (ou erro) e injeta no histórico.

4. Repeat: Com a nova observação, volta ao passo 1. Continua até: resposta final, max_steps ou erro fatal.

Exemplo real: "Reserve um voo barato para SP" → pesquisa (3 resultados) → compara preços → seleciona → confirma → responde ao usuário. 5 iterações.

Condições de parada (guardrails)

✅ Resposta final: o LLM decide que tem informação suficiente para responder.

🛑 max_steps: limite fixo (ex: 25 iterações). Previne loops infinitos.

💸 Orçamento: cap de tokens ou custo em dólar. Ex: "pare se gastar > $0.50".

💥 Erro fatal: tool retorna erro irrecuperável (API fora do ar, permissão negada).

⏰ Timeout: tempo máximo de execução total (ex: 60s).

Sem esses limites: em produção, um loop infinito pode custar milhares de dólares em minutos.

🏢 Frameworks: LangGraph (grafo de estados) · OpenAI Agents SDK (loop nativo) · Pydantic AI (tipado) · Smolagents (HuggingFace, minimalista) · LlamaIndex Workflows
💡 Insight: O loop que implementaremos no hands-on tem ~20 linhas em Python puro. O importante é entender a lógica, não decorar framework.

🎯 Deep dive: system prompt — a "constituição" do agente

O system prompt define quem o agente é, o que pode fazer e como deve se comportar. É o documento mais importante de todo o sistema.
Um bom system prompt responde 6 perguntas:
1. Quem é?
Identidade e papel
"Você é um auditor financeiro sênior"
2. O que quer?
Objetivo da missão
"Analise despesas e rejeite irregulares"
3. Com o que trabalha?
Ferramentas disponíveis
"Você tem acesso a: query_expenses, send_alert"
4. Como agir?
Procedimento e heurísticas
"Sempre verifique duplicatas antes de aprovar"
5. O que não pode?
Restrições e políticas
"NUNCA aprove > $500 sem supervisão humana"
6. Como reporta?
Formato de saída
"Retorne JSON: {approved: bool, reason: str}"
⚠️ Princípio: Trate o system prompt como código — versione em git, teste regressão, faça code review. Uma vírgula errada pode desbloquear comportamentos perigosos.
🏢 Referências reais: Os system prompts do Cursor, Devin, Claude.ai e v0.dev têm 2000-8000 palavras, são seccionados por função e evoluem semanalmente.

🎯 CO-STAR: framework para prompts robustos

Uma matriz prática para construir system prompts que funcionam. Preencha cada campo e o agente se comporta melhor.

C — Context
Cenário e informação de fundo
"Nossa empresa lança uma garrafa eco para Gen-Z"
O — Objective
Tarefa exata, sem ambiguidade
"Otimize esta descrição de produto para engajamento"
S — Style
Estilo de escrita ou execução
"Copywriter criativo de D2C"
T — Tone
Registro emocional
"Energético, autêntico, sem buzzwords"
A — Audience
Quem é o consumidor final
"Profissionais urbanos 22-30 anos"
R — Response
Formato e restrições da saída
"Tabela Markdown: original | revisão | razão"
💡 Por que funciona: Cada letra elimina uma fonte de ambiguidade. Sem CO-STAR, o LLM "adivinha" contexto, tom e formato — e frequentemente erra. Com CO-STAR, você programa o comportamento via linguagem natural.

CO-STAR na prática: construindo um agente auditor

SYSTEM_PROMPT = """
# Context
Você é o auditor de despesas de uma empresa de tecnologia com 200 funcionários.
O orçamento mensal de reembolso é R$50.000 e há histórico de fraudes.

# Objective  
Analise cada despesa submetida e aprove OU rejeite com justificativa.

# Style
Auditor financeiro sênior — rigoroso, metódico, baseado em evidências.

# Tone
Profissional e direto. Sem rodeios, sem "desculpe pelo inconveniente".

# Audience
Funcionários que submetem despesas e gestores que revisam decisões.

# Response
JSON: {"decisao": "aprovado|rejeitado", "valor": float,
       "justificativa": str, "regra_violada": str|null}

# Regras invioláveis
- Rejeitar qualquer despesa > R$50 sem nome de cliente E justificativa de negócio
- Rejeitar transporte > R$200 sem comprovante de trajeto
- NUNCA aprovar apenas porque o solicitante insiste ou pressiona
"""
✅ Cada seção do CO-STAR está mapeada. O agente sabe quem é, o que fazer, como se comportar e o que nunca fazer.

Anatomia de uma mensagem — os 4 papéis

Todo agente conversa em mensagens tipadas. Se você entender estas 4, já evita a maioria dos bugs.
system
Instruções do desenvolvedor.
{"role":"system","content":"Você é..."}
user
Input humano ou externo.
{"role":"user","content":"Qual o PIB?"}
assistant
Resposta do LLM ou pedido de tool.
{"role":"assistant","tool_calls":[...]}
tool
Resultado da ferramenta.
{"role":"tool","tool_call_id":"c1","content":"..."}
Fluxo central: assistant(tool_call) → tool(result) → assistant(...).

Estado: o que flui entre iterações

Diferente de uma chamada LLM solta, um agente carrega estado acumulado:

Estado explícito
Histórico, tool calls, resultados, scratchpad e system prompt.
Estado implícito
Iterações, custo, sessões abertas, cache e observabilidade.
Implicação prática: agentes não são "stateless" como uma API REST. Para escalar, persista o estado (Redis, banco) e torne o loop retomável — interrompido, deve poder continuar.

Autonomia: o espectro de controle

Quanto controle humano você dá ao agente?

Nível Descrição Exemplo
L0 — Suggest Agente propõe, humano executa tudo Copilot autocomplete
L1 — Confirm Cada ação requer aprovação Cursor "accept all" por arquivo
L2 — Bounded Autônomo dentro de limites (read-only, sandbox) Pesquisa web, análise de dados
L3 — Supervised Autônomo com revisão pós-fato PRs abertos por bots, drafts de email

🎯 Comece sempre em L1 ou L2. Subir de nível só depois de telemetria provando que o agente acerta em >95% dos casos do seu domínio.

⚠️ Onde os agentes quebram — falhas reais

Agentes em produção falham de formas previsíveis. Conhecer esses padrões antes de construir evita dias de debug.
🧠 LLM alucina argumentos de tool

O que acontece: O modelo inventa dados para preencher parâmetros obrigatórios. Ex: chama book_flight(destination="Marte") ou passa um ID que não existe.

Caso real: Um agente de suporte chamou a API de reembolso com um order_id inventado — processou reembolso para pedido errado.

Mitigação: JSON Schema estrito com enum para valores válidos + validação server-side antes de executar.

🛠️ Tool retorna resultado gigante (context overflow)

O que acontece: Uma busca SQL retorna 10.000 rows. O agente tenta colocar tudo no contexto → estoura a janela em 1 turno.

Impacto: Custo explode ($2-5 por chamada), latência sobe para 30s+, qualidade cai (informação relevante se perde no meio).

Mitigação: Paginar (LIMIT 10), resumir resultados antes de injetar, truncar com aviso ("mostrando 10 de 847 resultados").

🔄 Loop infinito (sem condição de parada)

O que acontece: Agent A envia arquivo para Agent B. B rejeita. A reprocessa errado e reenvia. Loop roda 500× em 2 min.

Custo real: Empresa reportou $4.700 gastos em 3 minutos por loop não detectado (API tokens + compute).

Mitigação: max_steps=25, watchdog de custo ($0.50 por execução), detecção de repetição (se último output = penúltimo → pare).

🎯 System prompt ambíguo → comportamento errático

O que acontece: Prompt diz "seja útil" mas não define limites. Agente aprova despesas irregulares, responde fora do escopo, ou muda de personalidade.

Exemplo: Agente de RH respondeu perguntas médicas porque o prompt dizia "ajude o funcionário em qualquer dúvida".

Mitigação: Definir explicitamente o que NÃO pode. Testar com adversarial prompts. Versionar e fazer A/B test de prompts.

🤝 Escolhe a tool errada (tool confusion)

O que acontece: Com 15+ tools disponíveis, o LLM confunde search_products com search_orders e retorna dados irrelevantes.

Por que acontece: Descrições das tools são parecidas ou vagas. O modelo não tem exemplos de quando usar cada uma.

Mitigação: Nomes e descrições disjuntas (sem sobreposição), incluir 1-2 exemplos de uso no schema, limitar a ≤7 tools por agente.

🔬 O que é RLHF? (Reinforcement Learning from Human Feedback)

O "segredo" do ChatGPT não foi só escala — foi ensinar o modelo a se comportar como humanos esperam.
A sigla significa: Reinforcement Learning (aprendizado por reforço) from Human Feedback (a partir de feedback humano). É uma técnica onde o modelo melhora iterativamente recebendo "notas" baseadas na preferência de avaliadores humanos.
LLM gera
2 respostas
→
👤 Humano escolhe
→
Reward Model
→
PPO otimiza
1. SFT (Supervised Fine-Tuning)
Humanos escrevem respostas-modelo. O LLM aprende o formato desejado (diálogo, instruções).
2. Reward Model
Treinado com milhares de comparações humanas ("A é melhor que B"). Aprende a dar "nota" a qualquer resposta.
3. PPO (Proximal Policy Optimization)
Algoritmo de RL que ajusta os pesos do LLM para maximizar a nota do Reward Model.

RLHF: antes e depois — exemplos concretos

O mesmo modelo, mesmos pesos — a diferença é apenas o alinhamento. Veja 3 cenários:
❌ Sem RLHF — "Recuse uma reunião"
"Dear Sir, I am writing to inform you that the meeting scheduled for Tuesday will not be attended by me. The reasons are varied..."
→ Genérico, verboso, impessoal
✅ Com RLHF — mesmo prompt
"Olá João! Não consigo terça. Reagendamos para quinta? Abraço, Alan"
→ Natural, direto, acionável
❌ Sem RLHF — "Como fazer bomba?"
"First, obtain potassium nitrate from fertilizer stores..."
→ Responde qualquer coisa sem filtro
✅ Com RLHF — mesmo prompt
"Não posso ajudar com isso. Se precisar de ajuda com química para fins educacionais..."
→ Recusa + redireciona
❌ Sem RLHF — "Resuma este artigo"
"The article discusses various topics including but not limited to the following areas of research..."
→ Enrolação, não resume nada
✅ Com RLHF — mesmo prompt
"O artigo propõe 3 pontos: (1) X resolve Y, (2) custo cai 40%, (3) limitação: não funciona em Z."
→ Estruturado, conciso, útil

1.4 Como o LLM "pensa"

Antes de construir agentes, vale revisar os 3 controles que mais mudam custo e comportamento:

🔤 Tokens

Unidade que o modelo enxerga. Português costuma gastar mais; você paga em input e output.

🪟 Context window

É a "mesa" onde cabem system, histórico e resultados. GPT-4o: 128k · Claude: 200k · Gemini: 1M+.

🎲 Temperature

0.0 = mais estável; 1.0 = mais criativo. Em produção, agentes ficam perto de 0.

Mental model: o LLM é uma função pura

f(prompt) → texto
⚠️ O LLM NÃO TEM MEMÓRIA entre chamadas.

Toda "memória" de um agente é reconstruída a cada chamada, concatenando o histórico inteiro no prompt.
Chamada 1:
system + user_msg_1 → resposta_1
Chamada 2:
system + user_msg_1 + resposta_1 + user_msg_2 → resposta_2
Isso explica por que o histórico longo fica caro e lento.

1.5 O padrão ReAct (Reason + Act)

📄 Yao et al., 2022 — "ReAct: Synergizing Reasoning and Acting in Language Models"

A ideia é simples e poderosa: fazer o LLM verbalizar o raciocínio antes de agir.

💭 Thought
→
⚡ Action
→
👁️ Observation
↺
✅ Final Answer
O loop repete até o modelo achar que tem a resposta. Esse padrão é a base de praticamente todos os agentes modernos — incluindo function calling que veremos no Encontro 2.

ReAct em ação — exemplo

Pergunta: "Qual a população do Brasil em milhões, multiplicada por 7?"

💭 Thought: Preciso primeiro descobrir a população do Brasil, depois multiplicar por 7.
⚡ Action: busca("população do Brasil")
👁️ Observation: "Aproximadamente 215 milhões (IBGE, 2024)."
💭 Thought: Agora multiplico 215 por 7.
⚡ Action: calculadora("215 * 7")
👁️ Observation: 1505
✅ Final Answer: Aproximadamente 1.505 milhões.

❓ Por que um agente forte usa a ferramenta errada?

Estudos da Anthropic sugerem que muitas falhas vêm de tools mal descritas, não do modelo.

📊 Onde agentes falham — distribuição de erros

Tool description ruim — 35%

Alucinação de tool — 10%
Parsing/formato — 10%
Outros — 5%
🎯 35% das falhas vêm de tools mal descritas — o problema mais comum NÃO é o modelo.

🛠️ Princípio crítico: design de ferramentas

❌ Ruim
def get_data(q: str) -> str:
    """Get data."""
Nome genérico, doc inútil, parâmetro sem contexto.
✅ Bom
def buscar_cliente_por_cpf(cpf: str) -> dict:
    """Busca dados cadastrais; retorna
    {nome, email, status}; use CPF
    com 11 dígitos sem pontuação;
    erros: 'NotFound' e 'Invalid'."""
Nome claro, contexto, formato de saída e erros explícitos.
A descrição da tool é parte do prompt. O modelo decide se e como usar baseado nela.

Os 7 mandamentos de tool design

1. Nome descritivo — verbo + objeto (enviar_email, não action1)
2. Docstring rica — o que faz, quando usar, formato dos args, erros possíveis
3. Argumentos tipados — use type hints + JSON Schema estrito
4. Erros amigáveis ao LLM — retorne "erro: CPF inválido, use 11 dígitos", não ValueError("x")
5. Output estruturado — JSON > prosa. O LLM consome melhor.
6. Poucas tools por agente — >15 começa a confundir. Use roteamento/skills.
7. Idempotência quando possível — chamar 2× = mesmo efeito de 1×. Protege contra loops.
💡 Teste prático: mostre a docstring para outra pessoa. Se ela conseguir usar a função corretamente sem ver o código, o LLM também conseguirá.

Parte 3: o loop do agente

📋 Colab-ready — requer o client da Célula 1
def run_agent(pergunta: str, max_steps: int = 6):
    msgs = [
        {"role": "system", "content": SYSTEM},
        {"role": "user",   "content": pergunta},
    ]
    
    for step in range(max_steps):
        resp = client.chat.completions.create(
            model="gpt-4o-mini", messages=msgs, temperature=0
        )
        out = resp.choices[0].message.content
        msgs.append({"role": "assistant", "content": out})
        
        if "Final Answer:" in out:
            return out.split("Final Answer:")[-1].strip()
        
        m = re.search(r"Action:\s*(\w+)\s*\nAction Input:\s*(.+)", out)
        if not m:  return "Agente não produziu ação válida."
        
        tool, arg = m.group(1).strip(), m.group(2).strip()
        obs = TOOLS.get(tool, lambda x: f"tool '{tool}' inexistente")(arg)
        msgs.append({"role": "user", "content": f"Observation: {obs}"})
    
    return "Máximo de passos atingido."

Observation: Aproximadamente 215 milhões (IBGE, 2024).

Action: calculadora
Action Input: 123 * 7 + 215
Observation: 1076
Final Answer: O resultado é aproximadamente 1.076 milhões.

O que observar ao rodar 👀

✅ O modelo verbaliza o "Thought" — isso é reasoning emergente. Ninguém ensinou explicitamente; ele aprendeu lendo a internet.
⚠️ Quando ele erra o formato (esquece "Action Input:" ou inventa), o loop quebra. Robustez vem de function calling estruturado (Encontro 2).
💸 Cada passo adiciona mais tokens ao contexto. Se a tarefa exige 10 passos, você paga 10× o histórico crescente. Esse é o início do problema de context management (Encontro 3).
🐛 Modelos pequenos (GPT-4o-mini, Llama 3.1 8B) às vezes ignoram o protocolo. Modelos maiores são mais consistentes. Isso é uma realidade prática importante.

Exercício 1.5 · The Strict Manager

Tarefa: implemente um system prompt usando CO-STAR que transforma o agente em um gerenciador de despesas rígido.

Contexto: você é gestor financeiro sênior de uma startup de 50 pessoas. O orçamento de reembolso é limitado, há tentativas recorrentes de fraude e você precisa de zero compromissos éticos.

Seu prompt deve conter:

  1. Context: empresa, tamanho, cenário de risco
  2. Objective: analisar despesas e decidir aprovação/rejeição
  3. Style: auditor implacável, sem empatia
  4. Tone: seco, profissional, sem desculpas
  5. Audience: funcionários e CFO
  6. Response: JSON estruturado com decisão binária + regras violadas

Teste seu prompt com 5 despesas reais ou fictícias:

  • Uma viagem "justificada" com hotel 5 estrelas
  • Uma refeição de trabalho com valor razoável
  • Um software/ferramenta sem OC prévio
  • Um reembolso pessoal (sim, alguém vai tentar)
  • Uma despesa legítima, dentro do policy

Entregáveis:

  1. Arquivo manager_prompt.txt com seu CO-STAR completo
  2. Arquivo test_cases.json com as 5 despesas + respostas do agente
  3. Nota (2-3 frases): qual foi a regra mais importante que seu prompt "entendeu"?
⚠️ Desafio adicional (opcional): tente fazer o agente "bugar" com uma despesa que pareça legítima, mas viola a lógica. Depois, corrija seu prompt para fechar o buraco.
