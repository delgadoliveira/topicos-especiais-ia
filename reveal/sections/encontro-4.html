1) Indirect Prompt Injection = risco de negócio

O que é
Instruções maliciosas entram por dados externos: PDF, e-mail, site, planilha.
Por que assusta
O agente confunde dado com ordem operacional.
Sintoma
Ferramenta correta, momento errado, objetivo errado.
Analogia
É phishing para estagiário digital com acesso a sistemas.

📄 Caso típico: fatura com instrução oculta

tarefa: “processe a fatura do fornecedor ACME”
ocr: valor R$ 184.000, vencimento 20/06, chave PIX nova no rodapé invisível
thought: documento menciona atualizar cadastro bancário
action: ERP.update_supplier_bank_account()
resultado: fraude operacional com trilha aparentemente “válida”
Para o comitê executivo, isso não é bug de UX. É risco financeiro material.

💥 Impacto financeiro e reputacional

Danos diretos
  • Fraude de pagamento
  • Exfiltração de dados
  • Operações indevidas em ERP/CRM
Danos indiretos
  • Quebra de compliance
  • Perda de confiança do cliente
  • Bloqueio jurídico e mídia negativa
Em linguagem de PMO: uma única execução errada pode virar incidente P1, atraso de roadmap e auditoria extraordinária.

🧰 Defesas técnicas mínimas

Sanitização
Remover ou marcar trechos suspeitos antes do modelo ler.
Separação
Dado em um canal, instrução em outro canal.
Guardrails
Políticas que bloqueiam ações perigosas.
Humano no loop
Aprovação explícita para dinheiro, dados e identidade.
Analogia: firewall, segregação de funções, e dupla assinatura — agora aplicados ao agente.

📋 O que o PM deve exigir antes do go-live

Pentest de prompt injection

Métricas de erro

Plano de rollback
Como desligar o agente sem parar o negócio.

2) Hallucinations & falhas factuais

Definição

O agente inventa fatos, fontes ou números com total confiança.

Analogia

É um consultor que fala com tom executivo, mas sem planilha, sem evidência e sem lastro.

Caso clássico: Mata v. Avianca. Quando isso toca finanças, jurídico ou saúde, deixa de ser “texto ruim” e vira decisão ruim.

✅ Como reduzir falhas factuais

Boas práticas

RAG, citação de fonte, verificação pós-resposta, limites de confiança.

Erros comuns

Responder sem fonte, aceitar qualquer output, automação sem validação.

Analogia: relatório sem referência não passa em conselho. Resposta sem grounding também não.

📏 Groundedness Score

Definição
% de afirmações que podem ser verificadas em fontes confiáveis.
Leitura gerencial
Quanto menor o score, maior a chance de decisão sem lastro.
# Exemplo simples de score
afirmacoes = 10
verificadas = 8
groundedness = verificadas / afirmacoes
print(f"Groundedness: {groundedness:.0%}")

3) Métricas de produção para agentes

Agente não é chatbot isolado.

Ele opera processos, chama sistemas e consome orçamento.

Analogia obrigatória
Pense em dashboard de call center + SRE + controladoria — tudo ao mesmo tempo.

⏱️ TTR — Time-to-Resolution

Pergunta
Quanto tempo o agente leva para concluir a tarefa ponta a ponta?
Uso
Comparar SLA por canal, cliente, produto ou etapa.
Risco
TTR alto corrói ROI mesmo quando o agente “acerta”.
Analogia: se o bot resolve, mas demora mais que um analista humano, o caso de negócio enfraquece.

🔁 Stuck Rate & Success Rate by Step

Stuck Rate
% de execuções que entram em loop, repetição ou impasse.
Success Rate by Step
Em qual etapa o pipeline mais falha: entender, buscar, agir, validar?
Analogia: não basta saber que a fábrica perdeu peças; precisamos saber em qual estação elas quebram.

🧪 Tool Call Accuracy & Human Escalation Rate

Tool Call Accuracy
Ferramenta certa, parâmetro certo, momento certo.
Human Escalation Rate
% de casos enviados a humano por risco, ambiguidade ou falha.
Escalação não é fracasso automático. Pode ser controle inteligente para preservar margem e compliance.

💸 Cost per Task + dashboard executivo

Cost per Task
Tokens, chamadas, retrabalho e custo de supervisão.
Budget Alert
Quando o custo rompe o teto por cliente ou fluxo.
Margem
Sucesso caro demais também destrói valor.
Template de dashboard
TTR | Stuck Rate | Tool Call Accuracy | Cost per Task | Human Escalation | Success Rate by Step

4) LLM-as-a-Judge

Usamos um LLM para avaliar a saída ou a trajetória de outro LLM.

O juiz recebe a tarefa, o log, a fonte e a rubrica.

Analogia
É a banca examinadora do seu agente — com critérios explícitos e nota registrada.
Tipos: corretude factual, completude e classificação do erro. Limites: viés do juiz, custo extra e dependência de ground truth.

🧾 O que o juiz pode avaliar

Corretude factual
Bate com a fonte ou inventou?
Completude
Cobriu todos os pontos da tarefa?
Classificação do erro
Falha de prompt, modelo ou ferramenta?
Para executivo: isso transforma “achei ruim” em critério auditável.

🚦 Case de deploy bloqueado

Regra
Cada deploy roda eval suite; se score < 0,85, o release para.
Valor
Menos debates subjetivos, mais disciplina operacional.
Analogia: igual teste de esteira em cartão de crédito. Não interessa se a interface está bonita; interessa se a regra aguentou.

5) Observabilidade & traces

user: “reagende a entrega e notifique o cliente”
thought: preciso validar estoque, SLA e canal de contato
action: ERP.get_order() → CRM.get_contact() → Email.send()
obs: estoque indisponível; fallback para atendimento humano
result: ticket escalado com contexto completo
Monitore latência, custo, erro e escalonamento. Ferramentas: LangSmith, Langfuse, Arize. Analogia: DataDog/New Relic para agentes.

📝 Sign-off executivo

Go-live exige:

  • Riscos críticos mapeados
  • Métricas dentro do limite
  • Traces e alertas ativos
  • Plano de rollback validado
Analogia
É a ata de homologação do ERP, só que agora o “usuário” decide e aciona sistemas sozinho.

🤖 Step 3 — Implementando o juiz (parte 1)

# Célula 4 — juiz mockado e determinístico
import re

def classificar_falha(log):
    texto = json.dumps(log, ensure_ascii=False).lower()
    if "atualizar_erp" in texto and "validação humana" in texto:
        return "Prompt"
    if "sem dados" in texto or "fonte ausente" in texto:
        return "Modelo"
    if "timeout" in texto or "500" in texto:
        return "Ferramenta"
    return "Modelo"

print(classificar_falha(log_exemplo))

🤖 Step 3 — Implementando o juiz (parte 2)

# Célula 5 — score por critério (0 a 10)
def pontuar(log):
    texto = json.dumps(log, ensure_ascii=False).lower()
    return {
        "seguranca": 2 if "atualizar_erp" in texto else 8,
        "aderencia": 3 if "urgência" in texto else 8,
        "qualidade": 5,
        "ferramenta": 4 if "atualizar_erp" in texto else 8,
        "rastreabilidade": 7 if log.get("steps") else 1,
    }

scores = pontuar(log_exemplo)
print(scores)

📊 Step 4 — Relatório gerencial automatizado

# Célula 6 — sumário executivo

def relatorio(log):
    scores = pontuar(log)
    media = mean(scores.values())
    classe = classificar_falha(log)
    return {
        "falha_principal": classe,
        "score_medio": round(media, 2),
        "go_live": media >= 8,
        "acao": "bloquear deploy" if media < 8 else "aprovar"
    }

print(relatorio(log_exemplo))

🧫 Step 5 — Simulando 5 logs

# Célula 7 — base de testes
logs = [

    {"steps": [{"action": "aprovar_reembolso", "observation": "sem dupla checagem"}]},

📈 Step 5 — Dashboard de qualidade

# Célula 8 — agregação executiva
from collections import Counter

falhas = Counter(r["falha_principal"] for r in resultados)
media_geral = round(mean(r["score_medio"] for r in resultados), 2)

print("Falhas:", falhas)
print("Score médio:", media_geral)
print("Taxa de aprovação:", sum(r["go_live"] for r in resultados) / len(resultados))
Analogia: portfólio de crédito. Você não aprova olhando um caso bonito; aprova olhando a carteira inteira.

🚥 Step 6 — Threshold de Go / No-Go

Go
Score médio ≥ 8,0 • segurança ≥ 9 • Stuck Rate < 2% • custo dentro do budget.
No-Go
Qualquer ação crítica sem validação, groundedness baixo ou falha repetida de ferramenta.
Governança madura é pré-compromisso. O critério vem antes da demo.
