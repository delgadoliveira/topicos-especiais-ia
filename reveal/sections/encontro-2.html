🧭 Vocabulário do dia — termos-chave

Os conceitos novos que você vai encontrar — uma frase cada. Clique em cada card para ver definição, exemplo e por que importa:

🧠 Reasoning model — modelo treinado para "pensar em voz alta" antes de responder.

O que é: gasta tokens internos ("thinking") raciocinando antes da resposta final, em vez de responder no impulso.

Exemplo: OpenAI o1/o3, DeepSeek R1, Claude extended thinking.

Por que importa: salto grande em matemática, código e planejamento — ao custo de mais tempo e tokens.

🪜 Chain-of-Thought (CoT) — pedir explicitamente "explique passo a passo".

O que é: instruir o modelo a verbalizar o raciocínio antes de dar a resposta final.

Exemplo: adicionar "mostre seu raciocínio" a qualquer prompt.

Por que importa: +20-40% em tarefas de raciocínio, com custo quase zero (Wei et al., 2022).

📋 Planning — plano completo antes de executar.

O que é: criar uma lista explícita de passos e dependências antes de agir. Como o roteiro antes de filmar.

Exemplo: Devin e Copilot Workspace mostram o plano antes de codar.

Por que importa: reduz loops e erros caros — e o humano pode revisar antes.

🔁 Reflexion — agente critica a própria resposta e tenta de novo.

O que é: tentar → avaliar (rodar testes, ler erros) → criticar a própria solução → tentar de novo com a crítica no contexto.

Exemplo: Cursor roda pytest, lê o traceback e corrige sozinho.

Por que importa: +20-30% em benchmarks de código (HumanEval), sem retreinar nada.

📦 Structured output — forçar a resposta em formato fixo (JSON).

O que é: obrigar o modelo a responder num esquema validável por código (JSON Schema / response_format).

Exemplo: extrair {nome, valor, data} de um e-mail de forma confiável.

Por que importa: sem formato fixo, a integração com sistemas quebra a cada resposta.

🛠️ Function Calling — o modelo retorna qual função chamar + args tipados.

O que é: em vez de texto livre, o modelo devolve tool_calls estruturados que seu código executa.

Exemplo: APIs da OpenAI/Anthropic com o campo tools — base de quase todo agente.

Por que importa: transforma intenção em ação executável e auditável.

Cada técnica que vamos estudar já está em produção em ferramentas que você talvez use:

🧠 Reasoning → ChatGPT o1/o3, Cursor

Como funciona: O modelo gasta 5-60 segundos "pensando" internamente (milhares de tokens invisíveis) antes de responder. Você vê "thinking..." na interface.

Caso real: ChatGPT o3 resolve problemas de matemática olímpica que GPT-4 errava. Cursor ativa reasoning mode quando detecta refatoração que toca 10+ arquivos.

Trade-off: 5-10× mais caro e lento. Ideal para planning, código complexo e decisões críticas — não para chat casual.

🪜 CoT → qualquer prompt, Wolfram Alpha

Como funciona: Basta adicionar "explique passo a passo" ou "mostre seu raciocínio" ao prompt. O modelo passa a verbalizar cada etapa antes de dar a resposta final.

Caso real: Google usa CoT internamente em seus agentes de busca. Wolfram Alpha sempre mostrou os passos — o conceito existia antes de ter nome.

Impacto medido: +20-40% de acurácia em tarefas de raciocínio (Wei et al., 2022). Custo: quase zero — só gasta mais tokens de output.

📋 Planning → Devin, Manus, Copilot Workspace

Como funciona: Antes de executar qualquer ação, o agente cria um plano explícito (lista de passos, dependências, checkpoints) e mostra ao usuário.

Caso real: Devin mostra um checklist antes de codar. Copilot Workspace gera um "specification → plan → implement" visível. Manus planeja a sequência de cliques no browser antes de iniciar.

Por que funciona: Reduz loops infinitos em ~60% (agente sabe o que falta), permite ao humano intervir antes de erros caros, e facilita debugging.

🔁 Reflexion → Cursor, Claude Code, AlphaCode

Como funciona: O agente tenta resolver → avalia o resultado (roda testes, lê erros) → critica a própria solução → tenta de novo com a crítica no contexto.

Caso real: Cursor gera código, roda pytest, se falhar lê o traceback e corrige automaticamente. Claude Code faz o mesmo com até 3 tentativas.

Impacto medido: +20-30% em benchmarks de código (HumanEval). Sem retreinar — apenas loop de autocrítica. O custo é 2-3× mais tokens por tarefa.

💡 Insight: Em 2025, quase todo produto agentic de sucesso combina 2 ou 3 dessas técnicas. Devin = Planning + Reflexion. Claude Code = CoT + Reflexion. Não é "escolha uma" — é "quando usar cada uma".

2.2 Chain-of-Thought (CoT)

📄 Origem
Wei et al. (2022): passos intermediários melhoram raciocínio.
🪄 Zero-shot
Kojima et al. (2022): “Let's think step by step”.
⚠️ Cuidado
Mais tokens; use para cálculo, lógica e decisão multi-etapa.
❌ Resposta direta
P: Maria tem 5 maçãs, dá 2 e compra 6. Quantas tem?

R: 11
O modelo pulou o estado intermediário: 5 - 2.
✅ Com Chain-of-Thought
1. Começa com 5
2. Dá 2 → fica com 3
3. Compra 6 → 3 + 6 = 9
O raciocínio cria checkpoints verificáveis.
Regra prática: CoT não é “pensar bonito”; é criar uma trilha que o humano, o teste ou outro modelo consegue auditar.

CoT em código

# Sem CoT
prompt_simples = "Quanto é 17 * 23 + 41?"

# CoT zero-shot
prompt_cot = """Quanto é 17 * 23 + 41?

Pense passo a passo antes de dar a resposta final."""

# CoT few-shot (mostre exemplos)
prompt_few_shot = """Resolva os problemas mostrando o passo a passo.

P: Quanto é 5 * 4 + 2?
R: 5*4 = 20. 20+2 = 22. Resposta: 22.

P: Quanto é 17 * 23 + 41?
R:"""

from transformers import pipeline
gen = pipeline("text-generation", model="Qwen/Qwen2.5-1.5B-Instruct", device_map="auto")
def llm(p):
    out = gen([{"role": "user", "content": p}], max_new_tokens=200)
    return out[0]["generated_text"][-1]["content"].strip()
def agente_reflexivo(tarefa, tentativas=3):
    critica = ""
    for _ in range(tentativas):
        p = f"Tarefa: {tarefa}\n"
        if critica: p += f"Crítica anterior: {critica}\nCorrija.\n"
        resp = llm(p + "Responda de forma direta:")
        aval = llm(f"Tarefa: {tarefa}\nResposta: {resp}\nEstá certa? 'OK' ou o erro.")
        if aval.upper().startswith("OK"): return resp
        critica = aval                 # vira contexto da próxima tentativa
    return resp
print(agente_reflexivo("Quanto é 17 * 24? Responda só o número."))
⚠️ Cuidado: gerar e criticar com o mesmo modelo tem viés (overconfidence). Em produção, use um modelo diferente ou maior como crítico.

📐 Structured Outputs

O problema não é “o modelo escrever bonito”. O problema é: outro sistema precisa consumir a resposta sem adivinhar formato.
1. Texto livre quebra automação
“João tem 30 anos” parece correto para humanos, mas para código vira regex frágil: acento, ordem, campo faltando e frase extra quebram o parser.
2. Schema vira contrato
Você define nome: str, idade: int, prioridade: 1..5. O LLM precisa preencher exatamente esse contrato.
3. Falha fica tratável
Se vier inválido, a aplicação sabe: rejeita, pede retry, manda para humano ou registra erro. Isso é produção.
class Pessoa(BaseModel):
    nome: str
    idade: int

pessoa = llm.with_structured_output(Pessoa).invoke("Extraia nome e idade")
# Pessoa(nome="João", idade=30)  → pronto para DB, API ou fila

🏗️ Pensamos bem — agora, como organizar a execução?

Até aqui, o foco foi qualidade do raciocínio (CoT, ToT, Reflexion). Mas um agente real não é só "pensar bem" — é coordenar prompts, tools, verificadores e até outros agentes na ordem certa.

❓ A pergunta nova
Quem faz o quê, e em que ordem? O que roda em paralelo? Onde colocar checagens? Quando parar?
💡 O ponto central
A maior parte dos agentes úteis não usa modelos mais espertos — usa orquestração melhor. Ex.: um fluxo de suporte que classifica → busca → responde → valida supera um único prompt gigante.
🗺️ Mapa da seção
1) Os 5 padrões canônicos (Anthropic 2024) → 2) árvore de decisão de qual usar → 3) Planning e plan-and-execute.
🍳 Analogia: reasoning é saber cozinhar bem um prato; orquestração é gerir a cozinha inteira de um restaurante — quem corta, quem frita, em que ordem, o que sai junto e quem confere o prato antes de servir.
Ideia-chave: antes de "autonomia", vem arquitetura de execução. Autonomia sem estrutura vira caos imprevisível.

Os 5 padrões agentic — visão geral

Catalogados pela Anthropic em "Building Effective Agents" (dez/2024), do mais previsível (1) ao mais autônomo (5). 💼 Caso único: um assistente de viagens corporativas — para cada padrão, os componentes e como implementá-lo nesse caso.

1. ⛓️ Prompt Chaining — etapas fixas; a saída de uma alimenta a próxima.
Componentes: sequência de prompts + gates de validação entre etapas.
💼 Viagem: extrair pedido → buscar voo → buscar hotel → montar itinerário, checando cada passo.
2. 🔀 Routing — classifica a entrada e despacha para o fluxo certo.
Componentes: um classificador (LLM/regra) + N rotas/handlers especializados.
💼 Viagem: separa "voo doméstico", "internacional" e "reembolso", cada um no seu fluxo.
3. ⚡ Parallelization — divide, roda em paralelo e agrega.
Componentes: fork (dividir) + execuções independentes + join/votação.
💼 Viagem: cota voo, hotel e carro ao mesmo tempo e junta tudo num orçamento.
4. 🧑‍💼 Orchestrator-Workers — um gerente decompõe e delega.
Componentes: orquestrador (decompõe em runtime) + workers + síntese final.
💼 Viagem: decide quantas cotações pedir por destino, delega a workers e consolida.
5. 🪞 Evaluator-Optimizer — gera, critica e refina em loop.
Componentes: gerador + avaliador (critério) + loop de melhoria.
💼 Viagem: monta itinerário → avalia política (preço/antecedência) → ajusta até passar.
📐 Regra prática
Um agente real combina 2–3 desses padrões. Comece pelo mais simples que resolve; só adicione autonomia quando o problema exigir. Detalhamos cada um a seguir.

Padrão 1 · Prompt Chaining — a linha de montagem

Origem teórica
Vem da lógica de pipelines da engenharia de software: dividir um trabalho complexo em etapas determinísticas. Em agentes, foi explicitado em Anthropic, “Building Effective Agents” (2024) e aparece na decomposição sequencial do AutoGen (Wu et al., 2023).

Problema que resolve
Quando um único prompt tenta fazer tudo, a qualidade cai: o modelo mistura objetivos, esquece restrições e produz saídas difíceis de validar. O chaining introduz focos locais e pontos de checagem intermediários.
Fundamento
Cada etapa reduz a incerteza da próxima. Em vez de pedir “entenda, traduza, resuma e formate”, o sistema transforma uma tarefa ampla em uma sequência com estado explícito.
Ideia-chave: cada passo pode usar prompt diferente, modelo diferente ou até código tradicional. Os gates capturam erro cedo, antes que ele contamine toda a pipeline.

Padrão 1 · Prompt Chaining — aplicação e arquitetura

Quando usar
Processo fixo e bem compreendido; a saída de uma etapa vira a entrada da próxima; você quer auditabilidade, logs e reexecução parcial.
Exemplos reais
Notion AI: destacar → traduzir → resumir → formatar.
Copilot Workspace: analisar issue → planejar mudanças → implementar → verificar testes.
Cenário forte
Jurídico: OCR → extração de entidades → classificação de cláusulas → score de risco. Cada fase pode ser validada com regras ou schemas.
Trade-off
Há mais latência porque são N chamadas em vez de 1, mas o ganho é confiabilidade, depuração mais fácil e mix de modelos: barato nas etapas simples, caro apenas nas difíceis.
Anti-pattern
Não use quando as etapas são fortemente interdependentes e precisam renegociar contexto o tempo todo. Nesse caso, a decomposição rígida atrapalha — prefira orchestrator.
Dica de implementação: trate cada etapa como função pura: input estruturado, output estruturado, validação e retry local. Isso deixa a pipeline observável e reprodutível.

Padrão 2 · Routing — aplicação e arquitetura

Critérios de decisão
Complexidade, domínio, urgência, sensibilidade a custo e nível de risco. O bom routing é, na prática, uma política de alocação de recursos.
Exemplo 1
Intercom/Zendesk: dúvida de cobrança → bot; risco de churn → escalonamento humano; pergunta técnica → bot especialista com KB.
Exemplo 2

Stripe Radar: transação simples → regra; suspeita → ML; alto risco → revisão humana.
Hint de implementação
Use prompt classificador com labels explícitos ou embeddings comparando a entrada com categorias protótipo. O resultado da rota deve ser logado para análise posterior.
Trade-off
O risco é o erro em cascata: rota errada → modelo errado → resposta ruim. Por isso, o classificador também precisa de avaliação própria.
Routing bem feito economiza não só dinheiro, mas também atenção humana: só os casos caros e arriscados sobem de nível.

Padrão 3 · Parallelization — aplicação e arquitetura

Sectioning na prática
E-mail triage: sentimento + prioridade + PII em paralelo.
Code review: estilo + bugs + segurança como análises independentes.
Perplexity: busca múltiplas fontes simultaneamente antes da síntese.
Voting na prática
Self-Consistency: 5 trilhas de raciocínio, voto majoritário; melhora matemática em ~10–20%.
AlphaCode: gera muitas soluções, testa e agrupa.
Código: 3 tentativas, fica a que passa todos os testes.
Arquitetura
O agregador precisa ser explícito: merge para sectioning, votação / rankeamento para voting. A latência final é max(tarefas), não a soma.
Trade-off
Custo tende a ser N×. No voting, prefira N ímpar e um bom mecanismo de agregação; sem isso, você paga por diversidade sem convertê-la em qualidade.
Heurística: paralelize quando os ramos não dependem entre si ou quando quer explorar múltiplas hipóteses antes de decidir.

Padrão 4 · Orchestrator-Workers — aplicação e arquitetura

Exemplo 1
Claude Code / Copilot Coding Agent: recebe uma issue, planeja arquivos, delega leitura/edição, roda testes e redistribui falhas para novos workers.
Exemplo 2
Deep Research: quebra a pergunta em subperguntas, delega buscas, sintetiza achados, identifica lacunas e manda pesquisar novamente.
Exemplo 3
Devin: orquestra escrita de código, comandos de terminal e ações de navegador como workers distintos sob um plano comum.
Quando escolher vs. parallelization
Quando as subtarefas não são conhecidas upfront, quando workers podem falhar e exigir retry, ou quando a síntese depende de ver todos os resultados antes de seguir.
Decisões de design
Quanta autonomia cada worker recebe? Como tratar falhas? Qual é o critério de parada? Sem essas políticas, o sistema vira um loop caro e difícil de controlar.
Hint de arquitetura: separe planner, executor e synthesizer. Mesmo quando tudo usa LLM, esses papéis ajudam a limitar contexto e responsabilidade.

Padrão 5 · Evaluator-Optimizer — aplicação e arquitetura

Exemplos reais
Cursor Tab: gera código → roda linter/testes → refina até ficar verde.
Writing assistants: rascunho → checagem de tom/clareza/gramática → melhoria.
Mais exemplos
AlphaCode/AlphaEvolve: gera, avalia com testes, seleciona e muta melhores candidatos.
Constitutional AI: gera → checa contra princípios → revisa.
Variantes
LLM-as-judge (barato), code-as-judge (determinístico) e human-in-the-loop (caro, mas verdade de chão).
Trade-off
O custo cresce como O(N×K), com N iterações e K tokens. Defina max_iterations, métricas de aceitação e uma condição de parada clara.
Risco
O avaliador também pode errar ou alucinar: “parece bom” não significa “está correto”. Sempre que possível, combine juiz probabilístico com sinais determinísticos.
Heurística prática: use esse padrão quando qualidade final importa mais que latência bruta — especialmente em código, escrita técnica e decisões auditáveis.

Planning — pensar antes de agir

Os dois pensam — a diferença é o que cada um pensa: o ReAct raciocina sobre o próximo passo (tático, local); o Planning raciocina sobre a tarefa inteira antes de gastar a primeira ação (estratégico, global).

🔁 ReAct — pensar para agir
🧠 O que pensa: "qual a melhor ação agora?" Decide 1 passo, observa o resultado e só então pensa no seguinte.
"Vou buscar opções."
tool: buscar_hotéis()
obs: 12 resultados → "e agora?"
⚠️ Limite: é míope — sem visão do todo, pode repetir ações, entrar em loop ou perder o objetivo em tarefas longas.
🗺️ Planning — pensar para planejar
🧠 O que pensa: "qual a rota inteira até o objetivo?" Gera o plano todo antes de executar, e revisa se algo desviar.
1) buscar 2) filtrar 3) comparar 4) justificar
executor roda cada etapa do plano
desvio? o planejador replaneja
✅ Ganho: visão global permite orçar, validar e paralelizar antes de gastar tools — e detectar inconsistências cedo.
💡 Por que Planning é evolução do ReAct: ele não substitui o ReAct — o envolve. O ciclo "agir → observar → decidir" do ReAct vira a fase de execução dentro de um plano maior, ganhando uma camada de previsão que o ReAct sozinho não tem.

ReAct ou Planning? — casos reais

Nenhum é "melhor": cada um brilha em um tipo de problema. O custo extra de planejar só compensa quando a tarefa é longa, cara ou arriscada.

🔁 Use ReAct quando…
…o caminho depende do que você descobre no meio — planejar tudo antes seria chute.
🌍 Casos reais:
• Suporte/atendimento: a próxima pergunta depende da resposta do cliente.
• Debugging: cada log muda a próxima hipótese.
• Pesquisa exploratória: "investigue X" — sem rota previsível, poucos passos.
🗺️ Use Planning quando…
…há muitos passos interdependentes, tools caras ou risco de retrabalho.
🌍 Casos reais:
• Viagem corporativa: orçamento, políticas e datas que se restringem entre si.
• Relatório multi-fonte: coletar, cruzar e sintetizar dados de várias APIs.
• Refatoração em vários arquivos: ordem importa; um passo errado quebra tudo.
Em uma frase: ReAct improvisa bem o curto prazo; Planning protege o longo prazo. Na prática, sistemas maduros combinam os dois — planejam a rota e usam ReAct para executar cada trecho.

Plan-and-Execute vs ReWOO

Ambos quebram a tarefa em um plano. A diferença central é quando o LLM vê os resultados das tools — e isso muda custo, velocidade e capacidade de adaptação.

🗺️ Plan-and-Execute
🔄 Como funciona: planeja → executa 1 passo → vê o resultado → replaneja. O LLM volta ao loop a cada etapa.
📞 Chamadas ao LLM: muitas (1 por passo + replan).
💸 Custo/tokens: alto — o contexto cresce a cada observação.
🛠️ Adapta a imprevistos: sim — corrige a rota ao ver erros ou dados novos.
✅ Melhor para: passos com dependência, ambiente que muda, erro provável.
🔀 ReWOO (Reasoning WithOut Observation)
🔄 Como funciona: planeja todos os passos de uma vez com variáveis (#E1, #E2…), sem ver resultados intermediários. Um Worker roda as tools; um Solver junta tudo no fim.
📞 Chamadas ao LLM: poucas (1 plano + 1 solver).
💸 Custo/tokens: baixo — não reenvia o contexto a cada passo (~5× menos tokens no paper).
🛠️ Adapta a imprevistos: não — o plano é fixo; se uma tool falha, não há replanejamento.
✅ Melhor para: coletas independentes/paralelizáveis, cenários sensíveis a custo e latência.
Em uma frase: o planner do Plan-and-Execute observa e se adapta; o do ReWOO é "cego" — planeja tudo antes e troca adaptação por menos custo e mais paralelismo.

🚫 Nenhum dos dois
Quando uma chamada simples já resolve — planejar só adiciona custo e latência.
🌍 Ex.: "qual a capital da França?"; um único lookup em banco; responder um FAQ direto.

🔄 Replanning — quando mudar a rota

Nenhum plano sobrevive ao primeiro contato com dados reais. Replanning é o ciclo Monitorar → Analisar → Planejar → Executar que transforma falhas em adaptação inteligente.

1. Monitorar
Após cada passo, o agente verifica: tool retornou erro? (timeout, 429, schema inválido), custo acumulado subiu além do budget? dados novos contradizem premissas iniciais?
Ex.: API de voos retorna 0 resultados → gatilho de replan.
2. Analisar
Distinguir falha local (um tool caiu, basta retry/fallback) de hipótese errada (a abordagem inteira não faz sentido). O agente usa o contexto acumulado para classificar: se 2+ passos seguidos falham no mesmo ponto, provavelmente a premissa está errada.
Técnica: self-reflection — "por que este passo falhou?"
3. Planejar
Duas estratégias: patch (trocar 1 passo — ex.: usar outra API) ou subplano novo (reescrever os passos restantes com nova abordagem). O custo de replanejar é real — cada replan consome tokens e aumenta latência.
Regra: patch se <30% dos passos afetados; subplano se >50%.
4. Executar
Retoma a execução mantendo trilha de auditoria: qual era o plano original, o que mudou, por quê. Isso é crítico para debugging e para o usuário entender o que aconteceu. Sem logs, replanning vira caixa-preta.
Formato: [replan #2] motivo: API timeout → ação: fallback para cache.
⚠️ Anti-padrão: Replan Loop
Replanejar a cada passo sem critério definido. O agente entra em loop infinito gastando tokens sem convergir. Solução: definir max_replans (ex.: 3) e gatilhos objetivos (ex.: só replaneja se erro ≥ 2x consecutivo ou custo > 80% do budget).
Caso real: agentes sem limite fizeram 15+ replans numa única query, custando $2+ sem resultado.
✅ Regra prática: Gatilhos antes, não depois
Defina critérios de replanning no momento do plano, não quando a falha acontece. Isso evita decisões ad-hoc sob pressão. Inclua no plano: replan_if: ["tool_error_2x", "budget_80pct", "conflicting_evidence"] e max_replans: 3.
Frameworks como LangGraph permitem definir esses gatilhos como edges condicionais no grafo.

def execute_with_replanning(objective: str, max_retries: int = 2):

🚀 Quatro upgrades modernos

Planejar ajuda muito. Mas, em tarefas difíceis, ainda dá para melhorar durante a execução.

⚡ Test-time compute

Gastar mais compute na inferência (não no treino). OpenAI o1/o3 usam isso: pensam mais antes de responder. Resultado: +30% em matemática vs GPT-4. Custo: 3-10x mais tokens.

🌳 LATS

Language Agent Tree Search: combina MCTS + LLM. Explora múltiplos caminhos, avalia cada um, aprofunda o melhor. Usado em HumanEval: +20% vs CoT simples. Trade-off: 5-50x mais chamadas.

✅ PRM

Process Reward Model: treina um modelo para avaliar cada passo do raciocínio. DeepMind usou em MATH: 78% → 92% de acerto. Essencial quando o resultado final pode estar "certo por acaso".

🏃 SPRINT

Speculative Planning with Reified Inference: planeja parcialmente → dispara ondas paralelas → usa resultado para replanejar. Reduz latência em 40-60% em tarefas com muitas tools. Inspiração: Cursor executa edits em paralelo enquanto planeja o próximo.

Function Calling estruturado

  1. Você descreve as tools com JSON Schema.
  2. O schema vai para a API no campo tools.
  3. O modelo devolve tool_calls já estruturados.
  4. Seu código executa a função real e devolve o resultado.
  5. O modelo sintetiza a resposta final.
🧪 No caso condutor: é assim que a "viagem para Porto Alegre" sai do raciocínio e vira ação — buscar_hotel(), buscar_voo() e calcular_orcamento() chamadas como tools tipadas e auditáveis.
É o mesmo espírito do ReAct, mas com contrato formal entre modelo e aplicação.

O que mudou em relação ao ReAct manual?

No ReAct manual, o LLM escrevia a ação em texto livre e o seu código tinha que ler com regex. O Function Calling (OpenAI, jun/2023) transformou isso num contrato nativo do modelo.

Antes — ReAct manual

🌍 No mundo real: a OpenAI lançou Function Calling em jun/2023; Anthropic (tool use) e Google (Gemini) seguiram. É o que faz o ChatGPT consultar sua agenda, o Zapier AI disparar ações e copilotos lerem CRM/banco de dados — tudo via contrato de tool, não texto solto.

🏭 Do artesanal para produção

Já temos raciocínio, planning e tool use. O problema agora não é "conseguir fazer uma demo" — é manter o sistema compreensível, observável e confiável quando ele encontra o mundo real.

🧪 Protótipo
Python puro, 2–3 tools, um loop while. Roda no notebook e impressiona na demo.
📌 Hackathon, POC interno, script pessoal.
📈 O que quebra na escala
10+ tools, múltiplos estados, APIs que falham de forma intermitente, prompts gigantes e custo descontrolado. E o pior: difícil saber onde o agente errou.
🏗️ Resposta da indústria

A passagem do artesanal para produção é menos sobre "modelo mais esperto" e mais sobre engenharia: estrutura, visibilidade e tolerância a falhas.

LangGraph — código + execução

from typing import Annotated, TypedDict
from langgraph.graph import StateGraph
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition

class State(TypedDict):
   messages: Annotated[list, add_messages]

llm = ChatOpenAI(model="gpt-4o-mini").bind_tools([calc, busca])

def agent(state: State):
   return {"messages": [llm.invoke(state["messages"])]}

graph = StateGraph(State)
graph.add_node("agent", agent)
graph.add_node("tools", ToolNode([calc, busca]))
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", tools_condition)
graph.add_edge("tools", "agent")
app = graph.compile()

result = app.invoke({"messages": [("user", "Quanto custa um hotel em SP?")]})

🧭 Dois filtros antes de produção

🔓 Open-source vs closed-source
Closed acelera protótipo e entrega modelos fortes com suporte. Open dá controle, custo previsível, privacidade e menos lock-in. A pergunta não é “qual é melhor?” — é qual restrição domina seu caso.
⚠️ Risco operacional
Prompt injection, vazamento via tool, custos em loop, autonomia excessiva e falha silenciosa. Quanto mais o agente age fora do chat, mais precisa de sandbox, allowlist, aprovação e logs.
Regra prática: prototipe com o melhor modelo disponível; produza com a combinação que maximiza controle + avaliação + custo aceitável.
