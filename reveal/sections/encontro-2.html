Aprovação humana
Pensar melhor ajuda. Ter processo é o que libera produção.

1) DAGs determinísticos

O que é
Um grafo rígido: cada nó é um passo definido; o caminho é previsível.
Valor executivo
Auditoria, repetibilidade, SLA e menor risco operacional.
Analogia: DAG = processo ISO 9001. Ninguém improvisa a checagem de qualidade.

2) Loops autônomos

O que é
O agente escolhe o próximo passo com base no que observa: ReAct, Reflexion, busca exploratória.
Valor executivo
Descoberta, criatividade, investigação em ambientes ambíguos.
Analogia: loop autônomo = brainstorming estratégico. Ótimo para explorar; péssimo para assinar contrato.

Quando usar cada abordagem?

DAG rígido

Compliance, onboarding KYC, crédito, seguros, aprovações financeiras, processos regulatórios.

Loop autônomo

Pesquisa de mercado, due diligence exploratória, ideação de produto, análise aberta de documentos.

Regra prática: quanto mais irreversível e regulado o ato, mais rígido deve ser o grafo.

📉 Caso realista de risco reputacional

Cenário
Uma fintech deixa o agente renegociar limites sem alçada humana em clientes de alta renda.
Consequência
Oferta inadequada, reclamação pública, revisão de controles e possível sanção regulatória.
Em finanças, autonomia total sem governança não parece inovação; parece multa da CVM.

🤖 Reasoning models & test-time compute

O que muda
Modelos como o1, o3 e DeepSeek R1 gastam mais compute em runtime para pensar melhor.
Implicação
Mais latência e custo por chamada; melhor precisão em tarefas realmente complexas.

Tabela executiva: qual modelo para qual tarefa?

TarefaModeloLatênciaCustoAccuracy
FAQ internomini/chat1sR$Boa
Triagem KYCchat + regras2sR$Muito boa
Análise crédito PJreasoning8sR$$$Superior
Due diligence M&Areasoning + humano20s+R$$$$Alta
Não pague “cérebros premium” para responder perguntas de FAQ.

Quando vale a pena pensar mais?

Vale a pena

Análise financeira complexa, cenários de M&A, detecção de inconsistências contábeis, priorização de risco.

Não vale a pena

FAQ de suporte, classificação trivial, preenchimento repetitivo, respostas que dependem só de regra fixa.

📊 Trade-off quantitativo: latência vs precisão

Modelo rápido
2s · R$0,02 · 87%
Reasoning
9s · R$0,18 · 93%
Delta
+6 pts por +7s e +9x custo
Pergunta de arquiteto: 6 pontos de precisão economizam mais do que custam?

Política de roteamento de modelos

Rota barata
Se risco baixo + resposta reversível → modelo rápido.
Rota premium
Se risco alto + impacto financeiro + ambiguidade → reasoning model + HITL.
Governança boa não é “sempre o melhor modelo”. É o modelo certo para o risco certo.

👥 Human-in-the-Loop (HITL)

Agente propõe
→
Humano aprova
→
Agente executa
Esse padrão converte autonomia em autonomia supervisionada.

Níveis de autonomia

Full auto
baixo risco
Semi-auto
notifica
Human approval
pausa e espera
Human only
IA só recomenda
Analogia: limites de cartão corporativo. Nem toda despesa merece a mesma liberdade.

Regras de escalação

Valor financeiro
Quanto maior a cifra, maior a alçada.
Risco reputacional
Uma ação pública errada vale mais que o ticket médio.
Irreversibilidade
Se desfazer é caro? Exija humano.
Boas regras de escalonamento são negócio puro, não “feature de IA”.

Exemplo corporativo de alçadas

< R$1K
auto-aprovado
R$1K–50K
notifica gerente
> R$50K
pausa + aprovação explícita
Clímax da aula: o agente só executa acima de R$50K depois do “sim” humano.

Congelar estado é parte da governança

Antes da pausa
Salvar contexto, decisão proposta, evidências e usuário responsável.
Depois da aprovação
Retomar exatamente do mesmo ponto, sem “reinventar” a análise.
Sem estado serializado, o agente reinterpreta. E reinterpretar pode gerar não conformidade.

    # Para aula, deixamos um mock reproduzível
    if usar_mock:
        if "120000" in prompt:

        model="gpt-4.1-mini",

    return resp.output_text

Step 1 — Definir estado do processo

def novo_estado(cliente, valor, renda_mensal):
    return {
        "cliente": cliente,
        "valor": valor,
        "renda_mensal": renda_mensal,
        "status": "em_analise",
        "parecer": None,
        "precisa_humano": False,
        "auditoria": [],
    }

estado = novo_estado("Empresa Atlas", 120000, 90000)

Step 2 — Agente analisa o pedido

def analisar_pedido(estado):
    prompt = (
        f"Analise crédito de {estado['cliente']} no valor de {estado['valor']} "
        f"com renda mensal de {estado['renda_mensal']}."
    )
    parecer = chamar_llm(prompt, usar_mock=True)
    estado["parecer"] = parecer
    estado["status"] = "decisao_proposta"
    estado["auditoria"].append("LLM gerou parecer inicial")
    return estado

estado = analisar_pedido(estado)

Step 3 — Regra de negócio HITL

def aplicar_regra_de_alcada(estado):
    if estado["valor"] > 50000:
        estado["precisa_humano"] = True
        estado["status"] = "aguardando_aprovacao_humana"
        estado["auditoria"].append("Pausa por alçada acima de R$50K")
    else:
        estado["status"] = "aprovado_automaticamente"
    return estado

estado = aplicar_regra_de_alcada(estado)
Aqui mora a governança: a regra não fica escondida dentro do modelo.

Step 4 — Congelar o estado

def congelar_estado(estado, caminho="estado_credito.json"):
    with open(caminho, "w", encoding="utf-8") as f:
        json.dump(estado, f, ensure_ascii=False, indent=2)
    return caminho

arquivo_estado = congelar_estado(estado)
print("Estado salvo em:", arquivo_estado)
Analogia: é como deixar a pasta do processo sobre a mesa do aprovador.

Step 5 — Simular aprovação humana

def pedir_aprovacao_humana():
    resposta = input("Aprovar operação acima de R$50K? (sim/nao): ").strip().lower()
    aprovador = input("Nome do aprovador: ").strip()
    return resposta, aprovador

resposta, aprovador = pedir_aprovacao_humana()
print(resposta, aprovador)
No produto real: botão no dashboard, fila em BPM ou task no Slack/Teams.

Step 6 — Retomar e executar

def retomar_e_executar(estado, resposta, aprovador):
    if resposta == "sim":
        estado["status"] = "executado"
        estado["auditoria"].append(f"Aprovado por {aprovador}")
        estado["auditoria"].append("Contrato enviado para assinatura")
    else:
        estado["status"] = "rejeitado_por_humano"
        estado["auditoria"].append(f"Negado por {aprovador}")
    return estado

estado = retomar_e_executar(estado, resposta, aprovador)

Step 7 — Registrar log de auditoria

def registrar_log(estado):
    momento = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    print("\n=== LOG DE AUDITORIA ===")
    print("cliente:", estado["cliente"])
    print("status:", estado["status"])
    print("quando:", momento)
    for item in estado["auditoria"]:
        print("-", item)

registrar_log(estado)

Demo final — o comportamento governável

obs: contrato enviado; log persistido
resultado: execução ocorreu com trilha, alçada e horário
Agora o agente não “toma poder”. Ele opera dentro da política.

Exercício 2 — impacto financeiro do erro

Se o agente erra 5% das vezes no nó de aprovação e cada erro custa R$20.000, qual o impacto esperado em 200 operações por mês?
200 operações
5% erro = 10 casos
10 × R$20K = R$200K/mês
É assim que se vende governança: mostrando o custo do “agente livre demais”.

Exercício 3 — CoT vs reasoning model

CoT + modelo barato

Mais rápido, ótimo para risco médio com regra clara e revisão humana eventual.

Reasoning model

Melhor para casos ambíguos, mas exige justificar o gasto adicional com ganho real.

Peça ao aluno: qual opção dá melhor custo-benefício para análise de risco da sua operação?

🧩 Takeaways executivos

Agentes úteis não bastam; agentes governáveis escalam.
DAGs protegem compliance; loops expandem descoberta.
Reasoning custa mais; use onde o erro custa ainda mais.
HITL acima de R$50K é padrão simples, poderoso e defensável.
