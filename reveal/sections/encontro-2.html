🧭 Vocabulário do dia — termos-chave

Os conceitos novos que você vai encontrar. Uma frase cada — aprofundamos durante a aula:

🧩 Você já viu isso em produtos reais

Cada técnica que vamos estudar já está em produção em ferramentas que você talvez use:

🧠 Reasoning → ChatGPT o1/o3, Cursor

Como funciona: O modelo gasta 5-60 segundos "pensando" internamente (milhares de tokens invisíveis) antes de responder. Você vê "thinking..." na interface.

Caso real: ChatGPT o3 resolve problemas de matemática olímpica que GPT-4 errava. Cursor ativa reasoning mode quando detecta refatoração que toca 10+ arquivos.

Trade-off: 5-10× mais caro e lento. Ideal para planning, código complexo e decisões críticas — não para chat casual.

🪜 CoT → qualquer prompt, Wolfram Alpha

Como funciona: Basta adicionar "explique passo a passo" ou "mostre seu raciocínio" ao prompt. O modelo passa a verbalizar cada etapa antes de dar a resposta final.

Caso real: Google usa CoT internamente em seus agentes de busca. Wolfram Alpha sempre mostrou os passos — o conceito existia antes de ter nome.

Impacto medido: +20-40% de acurácia em tarefas de raciocínio (Wei et al., 2022). Custo: quase zero — só gasta mais tokens de output.

📋 Planning → Devin, Manus, Copilot Workspace

Como funciona: Antes de executar qualquer ação, o agente cria um plano explícito (lista de passos, dependências, checkpoints) e mostra ao usuário.

Caso real: Devin mostra um checklist antes de codar. Copilot Workspace gera um "specification → plan → implement" visível. Manus planeja a sequência de cliques no browser antes de iniciar.

Por que funciona: Reduz loops infinitos em ~60% (agente sabe o que falta), permite ao humano intervir antes de erros caros, e facilita debugging.

🔁 Reflexion → Cursor, Claude Code, AlphaCode

Como funciona: O agente tenta resolver → avalia o resultado (roda testes, lê erros) → critica a própria solução → tenta de novo com a crítica no contexto.

Caso real: Cursor gera código, roda pytest, se falhar lê o traceback e corrige automaticamente. Claude Code faz o mesmo com até 3 tentativas.

Impacto medido: +20-30% em benchmarks de código (HumanEval). Sem retreinar — apenas loop de autocrítica. O custo é 2-3× mais tokens por tarefa.

💡 Insight: Em 2025, quase todo produto agentic de sucesso combina 2 ou 3 dessas técnicas. Devin = Planning + Reflexion. Claude Code = CoT + Reflexion. Não é "escolha uma" — é "quando usar cada uma".

2.1 Recap: por que o ReAct manual falha?

No Encontro 1 o agente funcionava porque o mundo era controlado. Em produção, a resposta do LLM vira entrada de outro sistema — e texto livre quebra.
1. Parser frágil

2. Sem contrato

O modelo pode inventar nome de tool, omitir campo ou mandar argumento no tipo errado.
3. Difícil recuperar
Erro → prompt maior
Erro → regex maior
Erro → mais exceções
Você conserta sintomas, mas não cria validação, retry nem observabilidade.

2.2 Chain-of-Thought (CoT)

📄 Origem
Wei et al. (2022): passos intermediários melhoram raciocínio.
🪄 Zero-shot
Kojima et al. (2022): “Let's think step by step”.
⚠️ Cuidado
Mais tokens; use para cálculo, lógica e decisão multi-etapa.
❌ Resposta direta
P: Maria tem 5 maçãs, dá 2 e compra 6. Quantas tem?

R: 11
O modelo pulou o estado intermediário: 5 - 2.
✅ Com Chain-of-Thought
1. Começa com 5
2. Dá 2 → fica com 3
3. Compra 6 → 3 + 6 = 9
O raciocínio cria checkpoints verificáveis.
Regra prática: CoT não é “pensar bonito”; é criar uma trilha que o humano, o teste ou outro modelo consegue auditar.

CoT em código

# Sem CoT
prompt_simples = "Quanto é 17 * 23 + 41?"

# CoT zero-shot
prompt_cot = """Quanto é 17 * 23 + 41?

Pense passo a passo antes de dar a resposta final."""

# CoT few-shot (mostre exemplos)
prompt_few_shot = """Resolva os problemas mostrando o passo a passo.

P: Quanto é 5 * 4 + 2?
R: 5*4 = 20. 20+2 = 22. Resposta: 22.

P: Quanto é 17 * 23 + 41?
R:"""

💡 Para agentes: o "Thought" do ReAct é CoT aplicado. Cada Thought é uma cadeia de raciocínio antes de cada Action.

📄 Wang et al., 2022
A ideia: gere N respostas com temperatura > 0 e use votação majoritária.

R: 9
Run 3
R: 11
Run 4
R: 9
Run 5
R: 9

🗳️ Voto: 4× "9" vs 1× "11"
↓

Trade-off: 5× mais caro e lento, mas sobe acurácia em ~10-20% em problemas com resposta única.
Quando usar: matemática, código, classificação — tarefas onde há uma resposta "certa" verificável.

🧠 Reasoning Models — o paradigma novo (2024+)

📅 OpenAI o1 (set/2024), o3 (dez/2024), DeepSeek R1 (jan/2025), Anthropic extended thinking (2025).

Em vez de "pensar via prompt", o modelo é treinado via RL para pensar muito antes de responder — gerando milhares de tokens internos invisíveis.
Modelos clássicos
Reasoning emergente via prompt (CoT). Latência típica: ~1s.
Reasoning models
Reasoning treinado; o modelo "pensa" 5–60s; matemática/código podem ganhar +30–50%.
⚠️ Trade-offs: 5–10× mais caro e 10–100× mais lento. Use para matemática, prova de teoremas, código complexo e planning multi-step — não para chat casual.

🪞 Self-Reflection & Reflexion

📄 Shinn et al., 2023 — "Reflexion: Language Agents with Verbal Reinforcement Learning"

A ideia: depois de tentar uma tarefa, o agente critica seu próprio output e tenta de novo com a crítica no contexto.

🎯 Tarefa
→
🤖 Tentativa 1
→
Sucesso?
→
🪞 Auto-crítica
'O que falhou?'
→
🤖 Tentativa 2
contexto + crítica
→
✅
📈 Resultado
Melhora 20-30% em HumanEval e ALFWorld, sem retreinar nada — apenas adicionando autocrítica ao loop.
💡 Analogia
Como um aluno revisando a prova antes de entregar: relê, encontra erros, corrige. Simples, mas eficaz.
🏢 Na prática: Cursor, Devin e Claude Code usam Reflexion internamente — quando o código não compila, o agente lê o erro, critica sua solução e tenta de novo.

Self-Reflection em código

📖 Em palavras (sem ler código): o agente tenta resolver, depois se critica ("isso de fato responde a pergunta?"), e se a crítica disser que não, ele tenta de novo levando a crítica em consideração. Repete até dar certo ou bater o limite de tentativas.

def agent_with_reflection(tarefa: str, max_tries: int = 3):
    historico = []
    for _ in range(max_tries):
        contexto = f"Tarefa: {tarefa}\n"
        if historico:
            contexto += f"Tentativas anteriores e críticas:\n{historico}\n"
        resposta = llm.invoke(contexto + "Sua resposta:")
        critica_raw = llm.invoke(f"""
        Tarefa original: {tarefa}
        Resposta proposta: {resposta}
        A resposta resolve a tarefa? Aponte erros específicos.
        Responda em JSON: {{"ok": bool, "criticas": [str]}}
        """)

Self-Reflection em código — continuação

        critica = json.loads(critica_raw)
        if critica["ok"]:
            return resposta
        historico.append({
            "tentativa": resposta,
            "criticas": critica["criticas"],
        })

    return resposta  # devolve melhor esforço

⚠️ Cuidado: usar o mesmo modelo pra gerar e criticar tem viés (overconfidence). Idealmente, use um modelo diferente — ou um maior — como crítico.

📐 Structured Outputs

O problema não é “o modelo escrever bonito”. O problema é: outro sistema precisa consumir a resposta sem adivinhar formato.
1. Texto livre quebra automação
“João tem 30 anos” parece correto para humanos, mas para código vira regex frágil: acento, ordem, campo faltando e frase extra quebram o parser.
2. Schema vira contrato
Você define nome: str, idade: int, prioridade: 1..5. O LLM precisa preencher exatamente esse contrato.
3. Falha fica tratável
Se vier inválido, a aplicação sabe: rejeita, pede retry, manda para humano ou registra erro. Isso é produção.
class Pessoa(BaseModel):
    nome: str
    idade: int

pessoa = llm.with_structured_output(Pessoa).invoke("Extraia nome e idade")
# Pessoa(nome="João", idade=30)  → pronto para DB, API ou fila
Nativo em OpenAI Structured Outputs, Anthropic tool use e Gemini. No Python, Pydantic AI e Instructor popularizaram esse padrão.

Pydantic AI — framework type-safe

📖 Em palavras: você descreve a forma do resultado e o framework devolve um objeto Python validado, pronto para DBs, filas e APIs — sem torcer para o JSON vir certo.
from pydantic import BaseModel; from pydantic_ai import Agent

class Ticket(BaseModel):
    titulo: str; prioridade: int; departamento: str; requer_resposta_imediata: bool

agent = Agent("openai:gpt-4o-mini", result_type=Ticket,
              system_prompt="Você classifica tickets de suporte.")
ticket = agent.run_sync("Cliente diz que servidor caiu, perdemos R$ 10k/h").data
print(ticket.prioridade, ticket.requer_resposta_imediata)  # 5 True
🎯 Por que importa: agentes em produção alimentam outros sistemas. Com schema, você troca parsing defensivo por validação explícita.

Planning brilha quando a tarefa tem múltiplos passos, custo relevante ou risco de retrabalho.

Plan-and-Execute vs ReWOO

🗺️ Plan-and-Execute
Planeja, executa, observa e corrige a rota.
Melhor quando o ambiente muda ou a etapa seguinte depende da anterior.
🔀 ReWOO
Planeja as evidências antes de chamar tools.
Melhor quando várias consultas podem ser preparadas e rodadas em paralelo.
Em uma frase: Plan-and-Execute lida melhor com dependência; ReWOO, com coleta paralela.

ReWOO

Solver(E1, E2, E3) -> recomendação
Escolha Plan-and-Execute
Quando há dependência ou erro provável.
Escolha ReWOO

Quando uma chamada simples já resolve.

def create_plan(objective: str) -> list[dict]:
   resp = client.chat.completions.create(
       model="Qwen/Qwen2.5-7B-Instruct",

           {"role": "system", "content": """

Responda APENAS JSON:
{"passos": [{"id": 1, "acao": "...", "tool": "busca|calculadora|escrita", "depende_de": []}]}

Fluxo mínimo: planejar → executar → detectar desvio → corrigir
def execute_with_replanning(objective: str, max_retries: int = 2):
   plan = create_plan(objective)
   for step in plan:
       result = execute_step(step)
       if result.failed and step.get("retries", 0) < max_retries:
           plan = replan(objective, plan, step, result.error)
           continue
   return synthesize(plan)

Gera várias respostas e escolhe a melhor.

Function Calling estruturado

  1. Você descreve as tools com JSON Schema.
  2. O schema vai para a API no campo tools.
  3. O modelo devolve tool_calls já estruturados.
  4. Seu código executa a função real e devolve o resultado.
  5. O modelo sintetiza a resposta final.
É o mesmo espírito do ReAct, mas com contrato formal entre modelo e aplicação.

Definindo tools

A descrição da tool é o “manual de uso” que o modelo lê para decidir quando e como chamá-la.
tools = [{
   "type": "function",
   "function": {
       "name": "calculadora",
       "description": "Avalia uma expressão matemática Python.",
       "parameters": {
           "type": "object",
           "properties": {
               "expr": {"type": "string", "description": "Ex: '2 + 3 * 4'"}
           },
           "required": ["expr"],
       },
   },
}]
Tool description ruim gera uso ruim de tool.

def run_agent_fc(pergunta: str, max_steps: int = 6):
   msgs = [
       {"role": "system", "content": SYSTEM_FC},
       {"role": "user", "content": pergunta},
   ]
   for _ in range(max_steps):
       resp = client.chat.completions.create(
           model="Qwen/Qwen2.5-7B-Instruct",
           messages=msgs,
           max_tokens=150,
       )
       out = json.loads(resp.choices[0].message.content)
       if "answer" in out:
           return out["answer"]
       msgs.append({"role": "assistant", "content": json.dumps(out)})

Exemplo em LangChain

O framework esconde o loop, o parsing e parte do error handling.
from langchain_openai import ChatOpenAI
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.tools import tool

@tool
def calculadora(expr: str) -> str:
   return str(eval(expr, {"__builtins__": {}}, {}))

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
prompt = ChatPromptTemplate.from_messages([
   ("system", "Você é um agente útil. Use as ferramentas quando precisar."),
   ("user", "{input}"),
   ("placeholder", "{agent_scratchpad}"),
])
agent = create_tool_calling_agent(llm, [calculadora], prompt)
executor = AgentExecutor(agent=agent, tools=[calculadora], verbose=True)

LangGraph — código + execução

from typing import Annotated, TypedDict
from langgraph.graph import StateGraph
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition

class State(TypedDict):
   messages: Annotated[list, add_messages]

llm = ChatOpenAI(model="gpt-4o-mini").bind_tools([calc, busca])

def agent(state: State):
   return {"messages": [llm.invoke(state["messages"])]}

graph = StateGraph(State)
graph.add_node("agent", agent)
graph.add_node("tools", ToolNode([calc, busca]))
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", tools_condition)
graph.add_edge("tools", "agent")
app = graph.compile()

result = app.invoke({"messages": [("user", "Quanto custa um hotel em SP?")]})

🎯 Boas práticas — antes de otimizar

Agora já vimos o ciclo completo: pensar, planejar, agir e orquestrar em produção. Falta o que mais diferencia um agente útil de um agente irritante: disciplina de projeto.

Analogia

Meta

Boas práticas — prompts + tools

✅ Faça
seja específico
dê contexto e restrições
quebre tarefas grandes
descreva tools com when/what/format
versione prompts como código
❌ Evite
prompt vago
contexto implícito
pedir tudo de uma vez
tool description genérica
confiar cegamente no output
Ferramenta boa tem escopo claro, parâmetros claros e erro tratável.

⚠️ Erros mais comuns

🔴
Loop infinito: repete a mesma ação. Fix: max_steps e detecção de repetição.
🔴
Tool alucinada: inventa ferramenta. Fix: validar nome antes de executar.
🔴
Overflow de contexto: conversa longa quebra. Fix: resumo e janela deslizante.
🟡
Custo explosivo: modelo caro em loop. Fix: budget cap e roteamento.
🟡
Resposta genérica: falta contexto. Fix: melhorar system prompt e exemplos.
Se o agente falha sempre, normalmente o defeito está no desenho do sistema, não no “QI do modelo”.

Exercício 2.4 · Plan-and-Execute na unha

def planner(pergunta: str) -> list[str]:
    """Pede ao LLM para gerar uma lista de passos."""
    ...

def executor(passo: str, contexto: list[str]) -> str:
    """Executa 1 passo, podendo usar tools."""
    ...

def plan_and_execute(pergunta: str):
    plano = planner(pergunta)        # ex: ["buscar X", "calcular Y", "resumir"]
    resultados = []
    for passo in plano:
        resultados.append(executor(passo, resultados))
    return resultados[-1]
