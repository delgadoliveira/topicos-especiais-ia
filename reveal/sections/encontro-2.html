🧩 Você já viu isso em produtos reais

Cada técnica que vamos estudar já está em produção em ferramentas que você talvez use:

🧠 Reasoning → ChatGPT o1/o3, Cursor

Como funciona: O modelo gasta 5-60 segundos "pensando" internamente (milhares de tokens invisíveis) antes de responder. Você vê "thinking..." na interface.

Caso real: ChatGPT o3 resolve problemas de matemática olímpica que GPT-4 errava. Cursor ativa reasoning mode quando detecta refatoração que toca 10+ arquivos.

Trade-off: 5-10× mais caro e lento. Ideal para planning, código complexo e decisões críticas — não para chat casual.

🪜 CoT → qualquer prompt, Wolfram Alpha

Como funciona: Basta adicionar "explique passo a passo" ou "mostre seu raciocínio" ao prompt. O modelo passa a verbalizar cada etapa antes de dar a resposta final.

Caso real: Google usa CoT internamente em seus agentes de busca. Wolfram Alpha sempre mostrou os passos — o conceito existia antes de ter nome.

Impacto medido: +20-40% de acurácia em tarefas de raciocínio (Wei et al., 2022). Custo: quase zero — só gasta mais tokens de output.

📋 Planning → Devin, Manus, Copilot Workspace

Como funciona: Antes de executar qualquer ação, o agente cria um plano explícito (lista de passos, dependências, checkpoints) e mostra ao usuário.

Caso real: Devin mostra um checklist antes de codar. Copilot Workspace gera um "specification → plan → implement" visível. Manus planeja a sequência de cliques no browser antes de iniciar.

Por que funciona: Reduz loops infinitos em ~60% (agente sabe o que falta), permite ao humano intervir antes de erros caros, e facilita debugging.

🔁 Reflexion → Cursor, Claude Code, AlphaCode

Como funciona: O agente tenta resolver → avalia o resultado (roda testes, lê erros) → critica a própria solução → tenta de novo com a crítica no contexto.

Caso real: Cursor gera código, roda pytest, se falhar lê o traceback e corrige automaticamente. Claude Code faz o mesmo com até 3 tentativas.

Impacto medido: +20-30% em benchmarks de código (HumanEval). Sem retreinar — apenas loop de autocrítica. O custo é 2-3× mais tokens por tarefa.

💡 Insight: Em 2025, quase todo produto agentic de sucesso combina 2 ou 3 dessas técnicas. Devin = Planning + Reflexion. Claude Code = CoT + Reflexion. Não é "escolha uma" — é "quando usar cada uma".

2.1 Recap: por que o ReAct manual falha?

No Encontro 1 fizemos um agente "na unha" com re.search. Funciona, mas escala mal.
😖 Problemas
  • Parsing frágil: regex quebra fácil
  • LLM alucina nomes de ferramentas
  • Argumentos viram strings mal formadas
  • Sem tipagem nem validação
  • Debug difícil quando algo falha
😎 Solução: Function Calling
  • OpenAI, Anthropic e Gemini suportam nativamente
  • JSON Schema valida argumentos
  • Modelo retorna estrutura, não texto livre
  • Permite múltiplas chamadas em paralelo
  • Virou padrão da indústria desde 2023
Antes das tools robustas, vamos entender como melhorar o reasoning.

🧠 Parte 1: Ensinando a pensar

Vamos ver 4 técnicas, da mais simples à mais sofisticada.

2.2 Chain-of-Thought (CoT)

📄 Wei et al., 2022 — "Chain-of-Thought Prompting Elicits Reasoning in LLMs"

A descoberta: pedir ao modelo para mostrar os passos melhora performance em 20–40% em tarefas de raciocínio.

❌ Sem CoT

P: Se Maria tem 5 maçãs e dá 2 para João, e depois compra mais 6, quantas tem?


R: 11 ❌

✅ Com CoT

P: ... Pense passo a passo.


R: Maria começa com 5. Dá 2 → fica com 3. Compra 6 → 3+6 = 9. Resposta: 9 ✅

🪄 A frase mágica: "Let's think step by step" (Kojima et al., 2022) — funciona em quase qualquer modelo, sem nenhum exemplo.

CoT em código

# Sem CoT
prompt_simples = "Quanto é 17 * 23 + 41?"

# CoT zero-shot
prompt_cot = """Quanto é 17 * 23 + 41?

Pense passo a passo antes de dar a resposta final."""

# CoT few-shot (mostre exemplos)
prompt_few_shot = """Resolva os problemas mostrando o passo a passo.

P: Quanto é 5 * 4 + 2?
R: 5*4 = 20. 20+2 = 22. Resposta: 22.

P: Quanto é 17 * 23 + 41?
R:"""

💡 Para agentes: o "Thought" do ReAct é CoT aplicado. Cada Thought é uma cadeia de raciocínio antes de cada Action.

🪞 Self-Reflection & Reflexion

📄 Shinn et al., 2023 — "Reflexion: Language Agents with Verbal Reinforcement Learning"

A ideia: depois de tentar uma tarefa, o agente critica seu próprio output e tenta de novo com a crítica no contexto.

🎯 Tarefa
→
🤖 Tentativa 1
→
Sucesso?
→
🪞 Auto-crítica
'O que falhou?'
→
🤖 Tentativa 2
contexto + crítica
→
✅
📈 Resultado
Melhora 20-30% em HumanEval e ALFWorld, sem retreinar nada — apenas adicionando autocrítica ao loop.
💡 Analogia
Como um aluno revisando a prova antes de entregar: relê, encontra erros, corrige. Simples, mas eficaz.
🏢 Na prática: Cursor, Devin e Claude Code usam Reflexion internamente — quando o código não compila, o agente lê o erro, critica sua solução e tenta de novo.

Self-Reflection em código

📖 Em palavras (sem ler código): o agente tenta resolver, depois se critica ("isso de fato responde a pergunta?"), e se a crítica disser que não, ele tenta de novo levando a crítica em consideração. Repete até dar certo ou bater o limite de tentativas.

def agent_with_reflection(tarefa: str, max_tries: int = 3):
    historico = []
    for _ in range(max_tries):
        contexto = f"Tarefa: {tarefa}\n"
        if historico:
            contexto += f"Tentativas anteriores e críticas:\n{historico}\n"
        resposta = llm.invoke(contexto + "Sua resposta:")
        critica_raw = llm.invoke(f"""
        Tarefa original: {tarefa}
        Resposta proposta: {resposta}
        A resposta resolve a tarefa? Aponte erros específicos.
        Responda em JSON: {{"ok": bool, "criticas": [str]}}
        """)

Self-Reflection em código — continuação

        critica = json.loads(critica_raw)
        if critica["ok"]:
            return resposta
        historico.append({
            "tentativa": resposta,
            "criticas": critica["criticas"],
        })

    return resposta  # devolve melhor esforço

⚠️ Cuidado: usar o mesmo modelo pra gerar e criticar tem viés (overconfidence). Idealmente, use um modelo diferente — ou um maior — como crítico.

📐 Structured Outputs

O problema não é “o modelo escrever bonito”. O problema é: outro sistema precisa consumir a resposta sem adivinhar formato.
1. Texto livre quebra automação
“João tem 30 anos” parece correto para humanos, mas para código vira regex frágil: acento, ordem, campo faltando e frase extra quebram o parser.
2. Schema vira contrato
Você define nome: str, idade: int, prioridade: 1..5. O LLM precisa preencher exatamente esse contrato.
3. Falha fica tratável
Se vier inválido, a aplicação sabe: rejeita, pede retry, manda para humano ou registra erro. Isso é produção.
class Pessoa(BaseModel):
    nome: str
    idade: int

pessoa = llm.with_structured_output(Pessoa).invoke("Extraia nome e idade")
# Pessoa(nome="João", idade=30)  → pronto para DB, API ou fila
Nativo em OpenAI Structured Outputs, Anthropic tool use e Gemini. No Python, Pydantic AI e Instructor popularizaram esse padrão.

Pydantic AI — framework type-safe

📖 Em palavras: você descreve a forma do resultado e o framework devolve um objeto Python validado, pronto para DBs, filas e APIs — sem torcer para o JSON vir certo.
from pydantic import BaseModel; from pydantic_ai import Agent

class Ticket(BaseModel):
    titulo: str; prioridade: int; departamento: str; requer_resposta_imediata: bool

agent = Agent("openai:gpt-4o-mini", result_type=Ticket,
              system_prompt="Você classifica tickets de suporte.")
ticket = agent.run_sync("Cliente diz que servidor caiu, perdemos R$ 10k/h").data
print(ticket.prioridade, ticket.requer_resposta_imediata)  # 5 True
🎯 Por que importa: agentes em produção alimentam outros sistemas. Com schema, você troca parsing defensivo por validação explícita.

🏗️ Os 5 padrões agentic da Anthropic

A referência da indústria · "Building Effective Agents" (Anthropic, 2024)

Até agora vimos como o LLM pensa. Agora vamos ver como organizar a execução — os padrões que as melhores equipes usam.

Os 5 padrões — visão geral

1. ⛓️ Prompt Chaining — passos fixos, output de um vira input do próximo. Simples e previsível.
2. 🔀 Routing — um classificador decide qual prompt/modelo especializado usar.
3. ⚡ Parallelization — tarefas independentes rodam em paralelo, depois agrega.
4. 🧑‍💼 Orchestrator-Workers — orquestrador dinâmico decompõe e delega para workers.
5. 🪞 Evaluator-Optimizer — gerador propõe, avaliador critica, em loop.

🎯 Lição central: a maioria dos "agentes" de sucesso são combinações desses 5 padrões — não loops autônomos puros.

Plan-and-Execute vs ReWOO

O dilema pedagógico: pensar a rota inteira reduz improviso, mas pode ficar caro e rígido. Dois padrões aparecem na literatura e nos frameworks.
🗺️ Plan-and-Execute
Planeja → executa → observa → replaneja.
Bom quando o ambiente muda: navegar web, debugar código, investigar dados. O executor aprende com cada observação antes de seguir.
🔀 ReWOO (Reasoning Without Observation)
Planeja todas as evidências antes de chamar tools.
Bom quando as consultas são independentes: coletar cotações, buscar fatos paralelos, comparar documentos.
Referências: ReAct (Yao et al., 2023), Plan-and-Solve prompting (Wang et al., 2023), ReWOO (Xu et al., 2023).

Plan-and-Execute vs ReWOO — cont.

Pergunta: "Compare laptops para ML até R$ 8k"

Plan-and-Execute:
  1. buscar opções → observar resultados
  2. filtrar GPUs → observar disponibilidade
  3. comparar preço/VRAM → responder

ReWOO:
  E1 = buscar("laptops RTX até 8k")
  E2 = buscar("benchmarks RTX 4060 laptop ML")
  E3 = buscar("preços varejo Brasil")
  Solver(E1, E2, E3) → recomendação
Use Plan-and-Execute
quando há dependência, erro provável ou decisão sensível.
Use ReWOO
quando evidências podem ser coletadas em paralelo.
Evite ambos
quando uma única chamada direta resolve.

🧠 Agentic Planning — como agentes criam e gerenciam planos

A ideia não nasceu com LLMs. Ela vem de planejamento em IA clássica: estado, ações, pré-condições, efeitos e objetivo.
🧱 Teoria clássica
STRIPS (Fikes & Nilsson, 1971) modelava o mundo como estado inicial → ações válidas → estado objetivo. HTN Planning quebrava metas em subtarefas hierárquicas.
🤖 O que muda com LLMs
O modelo não tem um simulador perfeito do mundo. Ele propõe um plano plausível, executa tools, observa evidências e corrige a rota.
Intuição: planning é a ponte entre “responder uma pergunta” e “gerenciar uma tarefa”.

📋 Anatomia de um plano de agente

Um bom plano é mais parecido com um contrato de execução do que com uma lista de tarefas.

🔄 Replanning — quando o plano precisa mudar

Replanning é um loop de controle: monitorar o mundo, detectar desvio e ajustar a rota. É próximo do ciclo MAPE-K da IBM Autonomic Computing.
Monitor
tool falhou? evidência chegou? custo subiu?
Analyze
falha local ou premissa do plano errada?
Plan
patch pequeno, subplano novo ou reinício total?
Execute
continua com trilha de auditoria.
Anti-padrão: replanejar a cada passo sem critério. Isso vira loop caro. Defina gatilhos objetivos.

🔄 Replanning — exemplos em produto

Devin / coding agents
Planejam tarefas de engenharia, executam testes, abrem PR e replanejam quando build ou lint falha.
Browser agents
Manus, Operator e afins precisam trocar rota quando UI muda, login bloqueia ou dado não aparece.
Claude Code / Copilot agents
Transformam plano em checklist: editar arquivo, rodar teste, interpretar erro, ajustar hipótese.
Regra prática
Peça plano explícito quando há 3+ passos, tools externas, custo alto ou risco de ação irreversível.
Leitura recomendada
Anthropic, Building Effective Agents (2024); LangGraph docs sobre persistence/checkpoints; Yao et al., ReAct.

import json

def create_plan(objective: str) -> list[dict]:
    """Pede ao LLM para decompor objetivo em passos ordenados."""
    resp = client.chat.completions.create(
        model="Qwen/Qwen2.5-7B-Instruct",
        messages=[
            {"role": "system", "content": "Decomponha o objetivo em 3-5 passos. Responda APENAS JSON: {\\"passos\\": [{\\"id\\": 1, \\"acao\\": \\"...\\", \\"tool\\": \\"busca|calculadora|escrita\\", \\"depende_de\\": []}]}"},
            {"role": "user", "content": objective},
        ],
        max_tokens=300,
        temperature=0.1,
    )
    plan = json.loads(resp.choices[0].message.content)["passos"]
    print(f"📋 Plano gerado para: {objective}")
    for step in plan:
        deps = f" (depende de {step['depende_de']})" if step["depende_de"] else ""
        print(f"   {step['id']}. {step['acao']} → tool: {step['tool']}{deps}")
    return plan

def execute_with_replanning(objective: str, max_retries: int = 2):
    plan = create_plan(objective)
    for step in plan:
        result = execute_step(step)
        if result.failed and step["retries"] < max_retries:

🚀 Além do planning: 4 upgrades modernos

Na trilha do Stanford CS329A, dá para deixar um agente mais forte na hora da execução sem trocar o modelo — só mudando como ele pensa, busca e verifica.

⚡ Test-time compute scaling

Em vez de 1 tentativa, você gasta mais compute na inferência: amostra várias respostas, compara, vota ou escolhe a melhor.

🎯 Analogia: não entregar a primeira redação do ENEM — fazer 5 rascunhos e enviar o melhor.
🌳 LATS

Usa busca em árvore, como motor de xadrez: explora caminhos alternativos, estima qual ramo parece promissor e segue por ele.

♟️ Não é “chutar o próximo lance”; é olhar algumas jogadas à frente.
✅ Verificação por passo

Em vez de checar só a resposta final, o sistema confere cada etapa do raciocínio. Isso reduz erros “bonitos, porém errados”.

🧮 Como um professor de matemática que corrige conta por conta, não só o resultado no fim.
🏃 SPRINT

Planeja um pouco, executa em paralelo, observa o que voltou e planeja de novo. Não espera um plano perfeito para começar.

🍳 Cozinha profissional: alguém corta legumes enquanto outra pessoa já deixa a panela no fogo.
Ideia central: mais pensamento útil em tempo de execução = agentes mais robustos em tarefas difíceis.

⚡ Test-time Compute Scaling

📚 Stanford CS329A destaca esta linha com Large Language Monkeys e Scaling LLM Test-Time Compute: gastar mais compute na inferência pode deixar o LLM mais “inteligente”.

Pergunta difícil
→

→
Amostra 2
→
Amostra 3...N
→
🗳️ voto / reranker / verificador
→
Melhor resposta
Repeated sampling

Faz várias tentativas com temperatura maior. Cada tentativa pensa de um jeito diferente.

Bom para: matemática, código, planejamento com múltiplos caminhos.

Best-of-N

Você gera N respostas e escolhe a melhor usando regras, voto, testes ou outro modelo avaliador.

Exemplo: gerar 5 soluções de código e ficar com a que passa mais testes.

Trade-off

Ganha qualidade, mas paga em latência e custo. Vale quando errar é caro.

💼 Fazer 1 orçamento é rápido; comparar 5 fornecedores costuma dar decisão melhor.
user: Qual arquitetura devo usar para um app com 10 milhões de usuários?
run 1: prioriza custo • run 2: prioriza simplicidade • run 3: prioriza resiliência
verifier: run 2 ignora escalabilidade; run 1 e 3 são válidos
selector: escolhe run 3 por cobrir escala + fallback
🏢 Em português: Gastar mais tempo pensando em problemas difíceis — como um aluno que relê a prova antes de entregar.

🌳 LATS — busca em árvore para agentes

📄 Zhou et al. — Language Agent Tree Search. A ideia é combinar reasoning + acting + planning com busca em árvore, como um motor de xadrez.

Objetivo
→
gera 3 próximos passos
→
avalia cada ramo
→
expande o melhor
→
executa / observa
→
melhor plano encontrado
ToT vs LATS

ToT explora vários pensamentos. LATS explora pensamentos e ações, usando feedback do ambiente para decidir qual ramo continuar.

Ou seja: é mais “agentic”.

Analogia de iniciante

Imagine planejar férias. Você abre 3 rotas: hotel no centro, perto do evento ou perto do aeroporto. Vai comparando preço, deslocamento e risco antes de escolher.

Quando usar

Tarefas com muitos caminhos possíveis: debugging difícil, web agents, quebra-cabeças, planejamento de viagem, pesquisa com várias fontes.

Quando é exagero

Responder “qual a capital do RS?” com busca em árvore seria desperdício. LATS é para cenários em que escolher o caminho é a parte difícil.

user: Monte um roteiro de 3 dias em Porto Alegre com foco em custo-benefício
ramo A: hotel central • ramo B: Airbnb mais afastado • ramo C: hostel premium
avaliação: B economiza mais, mas aumenta deslocamento; A tem melhor equilíbrio geral
expansão: no ramo A, agora compara restaurantes e transporte
🏢 Em português: O agente testa vários caminhos (como um GPS que calcula rotas alternativas) e escolhe o que deu certo.

✅ Verificação: ORM vs PRM

📄 “Let's Verify Step by Step” e a trilha de Robust Verification do Stanford CS329A mostram um ponto importante: às vezes o problema não é gerar mais — é conferir melhor.

ORM — Output Reward Model

Olha só o resultado final e pergunta: “parece bom?”

Vantagem: simples.

Limitação: pode aprovar uma resposta com passos internos errados.

📝 Como um professor que vê apenas “Resposta: 42”.
PRM — Process Reward Model

Avalia o processo passo a passo: esta soma faz sentido? esta premissa foi usada corretamente?

Vantagem: pega o erro cedo.

🧮 Como corrigir cada linha da conta antes do resultado final.
Passo 1
→
verifica ✓
→
Passo 2
→
erro detectado ✗
→
corrige antes de seguir
user: Se o hotel custa R$189/noite por 3 noites, quanto sobra de R$2000?
modelo: 189 × 3 = 547
PRM/verificador: multiplicação incorreta; o certo é 567
modelo corrige: 2000 - 567 = 1433
Resumo: ORM julga o placar final; PRM acompanha o jogo lance a lance.
🏢 Em português: Verificar só a resposta final (ORM) vs verificar cada passo do raciocínio (PRM). PRM pega erros antes.

🏃 SPRINT — planejar e executar ao mesmo tempo

📄 SPRINT = Enabling Interleaved Planning and Parallelized Execution in Reasoning Models. Em vez de “planejar tudo → executar tudo”, o modelo alterna planejamento e execução paralela.

Objetivo
→
plano parcial
→
execução A
→
execução B
→
novas observações
→
replaneja
→
próxima onda
1. Planeje o suficiente

Não espere o “plano perfeito”. Planeje só o necessário para disparar as primeiras subtarefas.

2. Execute em paralelo

Se duas buscas não dependem uma da outra, rode juntas: preços de hotel e previsão do tempo, por exemplo.

3. Replaneje com o que voltou

Os resultados parciais mudam o plano. Isso torna o agente mais rápido e mais adaptável.

🍽️ Analogia: um restaurante não espera a sobremesa ficar pronta para começar o prato principal. A cozinha coordena várias estações em paralelo e ajusta conforme os pedidos andam.

Quando brilha: tarefas longas com subtarefas independentes, como pesquisa na web, coleta de documentos, análise de logs e agentes de browser.

🏢 Em português: Em vez de planejar tudo antes, o agente planeja e executa ao mesmo tempo — como dirigir com o Waze recalculando em tempo real.

🧠 Modelos de Raciocínio vs Orquestração por Código

A consolidação de modelos de raciocínio nativo mudou como construímos agentes.
Antes (2023-2024)

Forçávamos modelos genéricos (GPT-4o, Claude Sonnet) a fazer ToT ou Plan-and-Execute via prompts complexos ou grafos no LangGraph.

O raciocínio era orquestrado pelo código: loops, condições, replanning — tudo externo ao modelo.

Agora (2025+)

Modelos de reasoning nativo (OpenAI o1/o3/o4-mini, DeepSeek R1, Gemini Flash-Thinking) pensam internamente antes de responder.

Padrão emergente: modelo de raciocínio gera o plano → modelo rápido com Function Calling executa as tools.

🎯 Quando gastar Test-time Compute?
Raciocínio interno (tokens de pensamento) é caro mas poderoso. Use para planejamento inicial e decisões complexas. Para execução repetitiva de tools, use modelos rápidos e baratos.

2.6 Function Calling estruturado

📅 Lançado pela OpenAI em junho/2023. Mudou o jogo.

Como funciona:

  1. Você descreve suas tools usando JSON Schema
  2. Manda no parâmetro tools=[...] da API
  3. O modelo retorna tool_calls com argumentos JSON validado
  4. Você executa as funções e devolve o resultado
  5. O modelo gera a resposta final (ou pede mais tools)

💡 É basicamente ReAct, mas o parsing/validação acontecem do lado da API, não na sua regex.

Function Calling — definindo tools

📖 Em palavras: essa lista é o "cardápio" do LLM: nome da função, descrição e parâmetros aceitos. É essa descrição que o modelo usa para decidir quando chamar cada tool.

tools = [{
    "type": "function",
    "function": {
        "name": "calculadora",
        "description": "Avalia uma expressão matemática Python.",
        "parameters": {
            "type": "object",
            "properties": {"expr": {"type": "string", "description": "Ex: '2 + 3 * 4'"}},
            "required": ["expr"],
        },
    },
}]
Para adicionar busca(query: str) ou outras tools, repita o mesmo schema como novos itens no array tools.

Function Calling — exemplo de chamada

Function Calling — o loop

📖 Em palavras: é o mesmo loop do Encontro 1, mas agora o LLM responde com um campo estruturado tool_calls em vez de texto bagunçado. Você (1) chama o modelo, (2) se ele não pediu ferramenta, é resposta final, (3) se pediu, você executa a função real, anexa o resultado e volta. Sem regex, sem parsing frágil.

def run_agent_fc(pergunta: str, max_steps: int = 6):
    msgs = [
        {"role": "system", "content": SYSTEM_FC},  # prompt com instruções JSON
        {"role": "user", "content": pergunta}
    ]
    for step in range(max_steps):
        resp = client.chat.completions.create(
            model="Qwen/Qwen2.5-7B-Instruct",
            messages=msgs, max_tokens=150,
        )
        out = json.loads(resp.choices[0].message.content)
        if "answer" in out:
            return out["answer"]
        # Modelo pediu uma ferramenta
        msgs.append({"role": "assistant", "content": json.dumps(out)})

O que mudou em relação ao ReAct manual

Antes (ReAct regex)
❌ Parsing manual de "Action:" / "Action Input:"
❌ Argumentos são strings
❌ Sem validação
❌ Quebra se o modelo "criar" ferramenta
❌ 1 tool por turno

✅ API entrega JSON estruturado
✅ Argumentos tipados
✅ Schema valida automaticamente
✅ Modelo só chama tools registradas

@tool
def calculadora(expr: str) -> str: return str(eval(expr, {"__builtins__": {}}, {}))
@tool
def busca(query: str) -> str: return {"capital do brasil": "Brasília"}.get(query.lower(), "Não encontrado")

2.9 LangGraph — controle total com grafos de estado

LangChain criou o LangGraph em 2024 como resposta à principal crítica: "abstrai demais, não sei o que está acontecendo".

Conceito: seu agente é um grafo dirigido onde cada nó é uma função Python e cada aresta é uma condição.

[*] --> agent: entrada do usuário
→
agent --> tools: precisa de tool?
→
agent --> END: resposta pronta
→
tools --> agent: resultado da tool
💡 Diferente do LangChain (loop implícito), aqui você desenha o fluxo — cada rota é explícita e debugável.
🏢 Adoção: empresas como Elastic, Replit, e bancos como JP Morgan usam LangGraph em produção por conta da auditabilidade.

LangGraph — exemplo completo comentado

from langgraph.graph import StateGraph
from langgraph.prebuilt import ToolNode, tools_condition
from langgraph.graph.message import add_messages
from langchain_openai import ChatOpenAI
from typing import Annotated, TypedDict

class State(TypedDict):
    messages: Annotated[list, add_messages]

llm = ChatOpenAI(model="gpt-4o-mini").bind_tools([calc, busca])

def agent(state: State):
    return {"messages": [llm.invoke(state["messages"])]}

# Monta o grafo
graph = StateGraph(State)
graph.add_node("agent", agent)
graph.add_node("tools", ToolNode([calc, busca]))
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", tools_condition)
graph.add_edge("tools", "agent")

app = graph.compile()

LangGraph — executando e observando

# Executa
result = app.invoke({
    "messages": [("user", "Quanto custa um hotel em SP?")]
})
print(result["messages"][-1].content)

# Streaming (token a token para UI)
for event in app.stream({"messages": [("user", "...")]}):
    print(event)  # mostra cada nó executado
✅ Vantagem: veja exatamente qual nó rodou, em que ordem, com que dados.

🎯 2.10 Boas Práticas — tirando o melhor dos agentes

Como interagir com agentes para maximizar resultados

📖 Analogia: prompts são como briefings para um estagiário brilhante. Ele é capaz, mas precisa de contexto claro, exemplos e limites.
🎯 Resultado esperado: um agente bem instruído resolve 80% dos casos sem intervenção. Mal instruído? Menos de 30%.

🎯 Boas práticas — prompts e contexto

✅ Faça
• seja específico
• dê contexto (stack, constraints, estilo)
• decomponha tarefas grandes
• itere em múltiplos turnos
• forneça exemplos few-shot
❌ Evite
• prompts vagos
• contexto implícito
• pedir tudo de uma vez
• ignorar erros do agente
• confiar cegamente no output

🛠️ Boas práticas — ferramentas e workflow

🔧 Ferramentas
• limite o escopo das tools
• nomeie bem
• documente params e ranges
• trate erros sem stack trace
• teste cada tool isoladamente
🔄 Workflow
• comece simples
• observe logs
• limite iterações
• monitore custo
• versione prompts como código

⚠️ Erros mais comuns (e como evitar)

🔴
Loop infinito: agente repete a mesma ação. Fix: max_steps + detecção de repetição.
🔴
Alucinação de tool: inventa ferramenta que não existe. Fix: validação estrita do nome antes de executar.
🔴
Context overflow: conversa longa estoura a janela. Fix: summarize + sliding window.
🟡
Custo explosivo: agente usa modelo caro em loop. Fix: budget cap + modelo menor para subtarefas.
🟡
Resposta genérica: ignora contexto. Fix: melhore o system prompt com exemplos.
🟡
Tool description ruim: modelo não entende quando usar. Fix: reescreva com when/what/format.

💡 Regra de ouro: se o agente falha repetidamente, o problema quase nunca é o modelo — é o design do sistema (prompt, tools, ou orquestração).

📊 Dado real: segundo a equipe do LangSmith, ~70% dos tickets de suporte de agentes são resolvidos melhorando tool descriptions ou system prompts — sem trocar o modelo.

Exercício 2.3 · LangGraph básico

Tarefa: monte o mesmo agente em LangGraph, seguindo o exemplo dos slides.

Adicione:

  • Um nó extra chamado validator que roda DEPOIS do agent mas ANTES de finalizar.

Exercício 2.4 · Plan-and-Execute na unha

Tarefa: implemente Plan-and-Execute sem framework. Estrutura:

def planner(pergunta: str) -> list[str]:
    """Pede ao LLM para gerar uma lista de passos."""
    ...

def executor(passo: str, contexto: list[str]) -> str:
    """Executa 1 passo, podendo usar tools."""
    ...

def plan_and_execute(pergunta: str):
    plano = planner(pergunta)        # ex: ["buscar X", "calcular Y", "resumir"]
    resultados = []
    for passo in plano:
        resultados.append(executor(passo, resultados))
    return resultados[-1]

Teste: "Compare o PIB do Brasil e da Argentina e diga em quantos % o do Brasil é maior."
